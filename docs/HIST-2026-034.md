# HIST-2026-034: Keycloak — Provisionamento de Contas para Motoristas e Passageiros

**Data**: 05/05/2026
**Status**: 📋 Planejado
**Tipo**: Security / Feature
**Tamanho**: Grande (4–5 dias)
**Prioridade**: Crítica

---

## 1. Contexto & Motivação

A revisão da integração Keycloak de maio/2026 revelou dois gaps críticos que tornam o ciclo de vida do usuário mobile incompleto e a revogação de acesso ineficaz.

### Gap 1 — Motoristas e passageiros não têm conta Keycloak criada pelo backend

O `SecurityConfiguration` protege os endpoints com:
```java
.requestMatchers("/api/v1/drivers/**").hasRole("DRIVER")
.requestMatchers("/api/v1/passengers/**").hasRole("PASSENGER")
```

Para acessar esses endpoints, o usuário precisa de um JWT com `ROLE_DRIVER` ou `ROLE_PASSENGER`. Essa role só existe se o usuário tiver uma conta no Keycloak com a role atribuída.

**O problema:** `CreateDriverUseCase` e `CreatePassengerUseCase` nunca chamam `IdentityProvisioningPort`. O backend cria o registro no PostgreSQL mas não cria a conta no Keycloak. O processo de auto-registro no Keycloak é implícito, não documentado e a atribuição de role é manual — dependência invisível de operação humana para cada motorista/passageiro.

Consequências diretas:
- Sem `keycloakId` salvo no banco, é impossível implementar `GET /me` (HIST-2026-031 e HIST-2026-033).
- Sem `keycloakId`, é impossível revogar a sessão de um usuário bloqueado.
- O processo de onboarding real tem um passo não documentado que pode ser perdido.

### Gap 2 — `disableUser` não existe no `IdentityProvisioningPort`

Quando `driver.block()` ou `passenger.block()` é chamado, apenas o banco de dados é atualizado. O Keycloak não é notificado. O token de acesso (`access_token`) permanece válido até expirar (padrão: 5 min) e o `refresh_token` pode ser renovado por até 30 minutos.

Na prática: um motorista banido por comportamento abusivo continua autenticado e pode completar corridas em andamento, aceitar novas corridas até o token expirar, e renovar o token até o `refresh_token` vencer.

## 2. Objetivo e Escopo

### 2.1 Provisionar conta Keycloak ao criar motorista ou passageiro

**Fluxo após esta história:**
```
POST /api/v1/drivers
  → CreateDriverUseCase
      → IdentityProvisioningPort.createDriverAccount(email, phone, tenantId)
          → Keycloak: cria user, atribui ROLE_DRIVER, seta tenant_id, retorna keycloakId
      → Driver salvo com keycloakId preenchido
      → DriverApproval criado em PENDING (HIST-2026-030)
```

**Novo método no `IdentityProvisioningPort`:**
```java
/**
 * Cria conta de motorista no Keycloak com ROLE_DRIVER.
 * O usuário criado não possui senha definida — autenticação é via fluxo de
 * "primeiro acesso" (e-mail de boas-vindas com link de definição de senha).
 *
 * @return keycloakId (UUID do usuário no Keycloak)
 */
String createDriverAccount(Email email, Phone phone, TenantId tenantId);

/**
 * Cria conta de passageiro no Keycloak com ROLE_PASSENGER.
 * Mesmo padrão: sem senha inicial, link de definição via e-mail.
 *
 * @return keycloakId (UUID do usuário no Keycloak)
 */
String createPassengerAccount(Email email, Phone phone, TenantId tenantId);
```

**Implementação no `KeycloakProvisioningAdapter`:**
- Cria `UserRepresentation` com `email`, `username = email`, `enabled = true`, `emailVerified = false`.
- Seta atributos: `tenant_id`, `phone`.
- Atribui role de realm: `ROLE_DRIVER` ou `ROLE_PASSENGER`.
- Adiciona ao grupo do tenant: `tenant:{tenantId}`.
- Envia e-mail de "definição de senha" via `usersResource.get(userId).executeActionsEmail(List.of("UPDATE_PASSWORD"))` — o usuário recebe um link para definir a própria senha sem que o backend a conheça.
- Retorna `keycloakId`.

**Campo `keycloakId` nas entidades:**

`Driver`:
```java
private final String keycloakId;  // nullable para compatibilidade com registros existentes
```

`Passenger`:
```java
private final String keycloakId;
```

Construtores de criação: `keycloakId` recebido como parâmetro e armazenado.
Construtores de reconstrução: incluir `keycloakId`.

**Alterações nos use cases:**
- `CreateDriverUseCase`: recebe `IdentityProvisioningPort` por construtor. Após salvar o driver, chama `identityProvisioning.createDriverAccount(...)` e atualiza o `keycloakId`. Se Keycloak falhar, faz rollback da criação do driver (operação dentro de `@Transactional`; a exceção propagada desfaz o `save`).
- `CreatePassengerUseCase`: mesmo padrão.

**Migrations Flyway:**
- `ALTER TABLE drivers ADD COLUMN keycloak_id VARCHAR(255)` (nullable).
- `ALTER TABLE passengers ADD COLUMN keycloak_id VARCHAR(255)` (nullable).

### 2.2 Implementar `disableUser` e `enableUser` no `IdentityProvisioningPort`

**Novos métodos no port:**
```java
/**
 * Desabilita o usuário no Keycloak imediatamente.
 * Sessões ativas expiram ao tentar renovar o token (refresh falha com 401).
 * O access_token atual continua válido até seu TTL natural (normalmente 5 min).
 */
void disableUser(String keycloakId);

/**
 * Reabilita o usuário no Keycloak (ex: após revisão administrativa).
 */
void enableUser(String keycloakId);
```

**Implementação no `KeycloakProvisioningAdapter`:**
```java
@Override
public void disableUser(String keycloakId) {
    UserRepresentation user = keycloakAdmin.realm(realm).users().get(keycloakId).toRepresentation();
    user.setEnabled(false);
    keycloakAdmin.realm(realm).users().get(keycloakId).update(user);
    // Revogar todas as sessões ativas (invalida refresh tokens imediatamente)
    keycloakAdmin.realm(realm).users().get(keycloakId).logout();
}
```

**Integração com `BlockDriverUseCase` (existente):**
- Após `driver.block()`, se `driver.getKeycloakId()` não for nulo, chamar `identityProvisioning.disableUser(keycloakId)`.
- `BlockDriverUseCase` passa a receber `IdentityProvisioningPort` como dependência opcional — se `keycloakId` for nulo (registros legados), apenas bloqueia no banco com log de aviso.

**Integração com `BlockPassengerUseCase` (a ser criado em HIST-2026-029):**
- Mesmo padrão.

**Integração com `ToggleTenantStatusUseCase` (HIST-2026-029):**
- Ao desativar um tenant, chamar `disableUser` para todos os admins do tenant no Keycloak. Para o MVP, aceita-se desabilitar apenas o admin principal; desabilitar todos os usuários do grupo requer iteração paginada no Keycloak Admin API.

## 3. Arquitetura e Modelagem

### 3.1 Domain

Alterações em `Driver` e `Passenger`:
- Campo `String keycloakId` (nullable, `final` — imutável após criação).
- Getter `getKeycloakId()`.
- Nenhum comportamento de domínio novo — o `keycloakId` é um identificador externo, não uma invariante de negócio.

### 3.2 Application

**`IdentityProvisioningPort`** — novos métodos:
- `createDriverAccount(Email, Phone, TenantId) → String`
- `createPassengerAccount(Email, Phone, TenantId) → String`
- `disableUser(String keycloakId)`
- `enableUser(String keycloakId)`

**Use cases alterados:**
- `CreateDriverUseCase` — recebe `IdentityProvisioningPort`.
- `CreatePassengerUseCase` — recebe `IdentityProvisioningPort`.

**Use cases que chamam `disableUser`:**
- `BlockDriverUseCase` (existente, via HIST-2026-029 ou ajuste direto).

### 3.3 Infrastructure

**`KeycloakProvisioningAdapter`** — implementar os 4 novos métodos.

**Flyway:**
- `V{next}__add_keycloak_id_to_drivers.sql`
- `V{next+1}__add_keycloak_id_to_passengers.sql`

**`UseCaseConfiguration`** — atualizar beans de `createDriverUseCase` e `createPassengerUseCase` para injetar `IdentityProvisioningPort`.

**Mappers JPA:**
- `DriverMapper`: mapear `keycloakId`.
- `PassengerMapper`: mapear `keycloakId`.

### 3.4 Presentation

Nenhuma mudança em controllers ou DTOs — `keycloakId` é detalhe interno, não exposto na `DriverResponse` ou `PassengerResponse`.

## 4. Critérios de Aceite

### Provisionamento
- [ ] `POST /api/v1/drivers` cria usuário no Keycloak com `ROLE_DRIVER`, grupo `tenant:{tenantId}`, atributo `tenant_id`. Retorna 201 com `id` do driver.
- [ ] Driver criado no banco tem `keycloakId` preenchido com o UUID Keycloak.
- [ ] Se Keycloak estiver indisponível, `POST /api/v1/drivers` retorna 503 e nenhum registro é criado no banco (rollback).
- [ ] Usuário Keycloak recebe e-mail de definição de senha após criação.
- [ ] `POST /api/v1/passengers` segue o mesmo padrão com `ROLE_PASSENGER`.

### Desabilitação
- [ ] Ao bloquear um motorista (`block()`), se `keycloakId` estiver preenchido, o usuário é desabilitado no Keycloak e suas sessões ativas são revogadas.
- [ ] Após bloqueio, tentativa de `refresh_token` pelo motorista retorna 401 (sessão revogada).
- [ ] Se `keycloakId` for nulo (registros legados), o bloqueio no banco ocorre normalmente com log de aviso — sem falha.
- [ ] `enableUser` reabilita a conta Keycloak (para futuro fluxo de desbloqueio administrativo).

### Geral
- [ ] `mvn test` verde — ArchUnit + testes unitários com mock de `IdentityProvisioningPort`.
- [ ] `mvn verify` verde — IT usa `MockIdentityProvisioningPort` (sem Keycloak real no Testcontainers). Verificar se Testcontainers tem imagem Keycloak configurada; se não, usar mock.
- [ ] Registros de driver/passageiro existentes (com `keycloakId = null`) continuam funcionando sem quebrar.
