# HIST-2026-029: Backend Administrativo e Dashboards (Admin API)

**Data**: 02 de Maio de 2026  
**Status**: 📋 Planejado  
**Tipo**: Feature / Plataforma Administrativa  
**Tamanho**: Grande (5-7 dias)  
**Prioridade**: Alta

---

## 1. Contexto & Motivação

Atualmente, o RappiDrive possui funcionalidades robustas para o fluxo principal (motoristas, passageiros, corridas, pagamentos) e um processo automatizado para criar novos tenants (Cidades/Operadores) via `SuperAdminController`. Além disso, existe o fluxo de aprovação de motoristas recém-cadastrados (HIST-2026-012).

No entanto, falta uma **camada de gestão operacional completa** para o dia a dia da plataforma. Tanto o dono da plataforma (Super Admin global) quanto as equipes de operação locais (Admins do Tenant) operam "às cegas" após o onboarding inicial.

### Por que é crítico?
- **Gestão de Operadores (Tenants):** O Super Admin não consegue listar, editar ou desativar operadores existentes.
- **Gestão de Qualidade:** Admins locais não conseguem visualizar a lista completa de motoristas/passageiros, analisar histórico ou banir contas abusivas proativamente.
- **Resolução de Conflitos:** O suporte não consegue investigar corridas problemáticas ou analisar reembolsos e contestações sem acesso direto ao banco de dados.
- **Business Intelligence:** Faltam endpoints para alimentar dashboards com métricas operacionais (corridas do dia, faturamento, motoristas ativos).

---

## 2. Objetivo e Escopo

Implementar uma suíte de APIs administrativas dividida em dois níveis de acesso, respeitando o modelo de segurança e isolamento multi-tenant (Keycloak + Hibernate Filters) já estabelecido.

### 2.1. Super Admin API (Gestão Global da Plataforma)
Foco exclusivo na gestão das franquias/cidades.
*   **Listar Tenants:** `GET /api/admin/tenants` (com paginação e filtros de status).
*   **Detalhar Tenant:** `GET /api/admin/tenants/{id}` (informações completas + métricas macro).
*   **Atualizar Tenant:** `PUT /api/admin/tenants/{id}` (alterar nome, email admin local).
*   **Desativar Tenant:** `PATCH /api/admin/tenants/{id}/deactivate` (soft-delete ou suspensão de operação, impedindo novos logins/corridas).
*   **Reativar Tenant:** `PATCH /api/admin/tenants/{id}/activate`.

### 2.2. Tenant Admin API (Gestão Operacional Local)
Foco na operação diária de uma cidade específica. As rotas abaixo utilizarão o filtro automático de tenant (`X-Tenant-ID`).

#### Gestão de Usuários (Motoristas e Passageiros)
*   **Listar Motoristas:** `GET /api/v1/admin/drivers` (filtros: status, busca por nome/email/CPF).
*   **Bloquear/Banir Motorista:** `PATCH /api/v1/admin/drivers/{id}/block` (por infração, com motivo).
*   **Histórico do Motorista:** `GET /api/v1/admin/drivers/{id}/history` (resumo de corridas e ganhos).
*   **Listar Passageiros:** `GET /api/v1/admin/passengers`.
*   **Bloquear/Banir Passageiro:** `PATCH /api/v1/admin/passengers/{id}/block`.

#### Gestão e Monitoramento de Corridas
*   **Monitoramento em Tempo Real:** `GET /api/v1/admin/trips/active` (retorna corridas `REQUESTED`, `ASSIGNED`, `IN_PROGRESS` para o mapa do dashboard).
*   **Histórico de Corridas:** `GET /api/v1/admin/trips` (filtros por data, status, motorista, passageiro).
*   **Detalhes da Corrida (Admin View):** `GET /api/v1/admin/trips/{id}` (visão estendida, incluindo logs de eventos, falhas de pagamento, etc.).
*   **Forçar Cancelamento:** `POST /api/v1/admin/trips/{id}/force-cancel` (ação de suporte em caso de emergência ou app travado).

#### Métricas do Dashboard (Analytics MVP)
*   **Métricas Diárias:** `GET /api/v1/admin/analytics/daily` (total de corridas, motoristas online, faturamento bruto estimado).

## 3. Arquitetura e Modelagem

### 3.1. Domain Layer
A maior parte do domínio já existe, mas precisaremos de pequenos ajustes:
*   **Tenant Entity:** Adicionar comportamentos `deactivate()` e `activate()`. Validar transições de estado para evitar desativar um tenant já inativo.
*   **Driver & Passenger Entities:** Já possuem o método `block()`. Será necessário garantir que o motivo do bloqueio (opcional) seja registrado, possivelmente via Domain Event.
*   **Trip Entity:** Adicionar `forceCancel(AdminId, Reason)` que ignora políticas de taxa (pois é uma ação administrativa de suporte).

### 3.2. Application Layer (Use Cases)

**Novos Use Cases (Super Admin):**
*   `ListTenantsUseCase`
*   `GetTenantDetailsUseCase`
*   `UpdateTenantUseCase`
*   `ToggleTenantStatusUseCase` (Activate/Deactivate)

**Novos Use Cases (Tenant Admin):**
*   `SearchDriversUseCase` (Paginação + Especificações de busca flexíveis)
*   `BlockDriverUseCase` (Invoca `driver.block()`, publica evento, opcionalmente derruba sessão no Keycloak)
*   `SearchPassengersUseCase`
*   `BlockPassengerUseCase`
*   `MonitorActiveTripsUseCase`
*   `SearchTripsUseCase`
*   `ForceCancelTripUseCase`
*   `GetDailyAnalyticsUseCase`

### 3.3. Infrastructure Layer
*   **Tenant Repository:** Adicionar métodos de paginação e busca no `SpringDataTenantRepository`.
*   **Queries Dinâmicas:** Implementar `Specification` do Spring Data JPA ou `CriteriaBuilder` para as buscas complexas de motoristas e corridas (ex: buscar motorista pelo começo do nome E status = ACTIVE).
*   **Analytics:** Para o `GetDailyAnalyticsUseCase`, criar queries nativas otimizadas (ou JPQL com agregações) no `JpaTripRepositoryAdapter` para evitar carregar todas as corridas na memória ao calcular faturamento e contagens.

### 3.4. Presentation Layer (REST API)
*   **Controllers:** `AdminTenantController` (Super Admin), `AdminDriverController`, `AdminPassengerController`, `AdminTripController`, `AdminAnalyticsController` (Tenant Admin).
*   **DTOs:** Criação de diversos DTOs de `Response` focados na visão administrativa (ex: `AdminTripDetailResponse` contendo dados que o passageiro normalmente não vê).
*   **Segurança:**
    *   Super Admin API: `@PreAuthorize("hasRole('SUPER_ADMIN')")`.
    *   Tenant Admin API: `@PreAuthorize("hasAnyRole('SUPER_ADMIN', 'COMPLIANCE_OFFICER', 'SUPPORT_ADMIN')")` variando conforme a criticidade (ex: `SUPPORT_ADMIN` pode ver, mas não pode usar o `forceCancel`).

## 4. Integração Keycloak (Ações Administrativas)

Quando um Admin bane um usuário (Motorista/Passageiro) ou desativa um Tenant, a ação não deve ocorrer apenas no banco relacional da aplicação.

**Bloqueio de Usuário:**
*   O `BlockDriverUseCase` deve chamar o `IdentityProviderPort.disableUser(keycloakUserId)`.
*   Isso garante que o usuário seja imediatamente desconectado e impedido de renovar o token (`refresh_token`).

**Desativação de Tenant:**
*   Idealmente, todos os usuários do grupo do tenant no Keycloak devem ter o acesso revogado, ou os clientes daquele realm devem ser desabilitados temporariamente. Para o MVP, a desativação no banco de dados (`active = false`) bloqueada no `TenantResolverFilter` já é suficiente para barrar o acesso HTTP à API.

## 5. Critérios de Aceite

### Funcionais
- [ ] Super Admin consegue listar todos os tenants de forma paginada.
- [ ] Super Admin consegue desativar um tenant, e requisições subsequentes para este tenant usando `X-Tenant-ID` retornam `403 Forbidden` ou `401`.
- [ ] Tenant Admin consegue listar motoristas e passageiros com paginação e busca por nome/email.
- [ ] Tenant Admin consegue bloquear um motorista; o motorista não consegue mais aceitar corridas e tem sua conta Keycloak desabilitada.
- [ ] Tenant Admin visualiza painel de corridas ativas (em andamento ou aguardando motorista).
- [ ] Tenant Admin consegue forçar o cancelamento de uma corrida problemática sem gerar cobrança para as partes.
- [ ] Endpoint de Analytics retorna total de corridas do dia, faturamento e motoristas online.

### Técnicos & Segurança
- [ ] Nenhuma violação do isolamento multi-tenant (Admins do Tenant A não veem motoristas do Tenant B). Garantido via `TenantFilterAspect`.
- [ ] Ações destrutivas (`force-cancel`, `block`) geram eventos de domínio para auditoria (Outbox Pattern).
- [ ] Queries de busca e listagem utilizam paginação (`Pageable`) para evitar problemas de memória (OOM).
- [ ] Endpoints documentados com Swagger/OpenAPI.
- [ ] Regras de arquitetura Hexagonal validadas com sucesso no ArchUnit.
