# HIST-2026-033: UX Flow — Endpoint Me, Pagamento no Onboarding e Endereços Salvos

**Data**: 05/05/2026
**Status**: 📋 Planejado
**Tipo**: Feature / UX
**Tamanho**: Médio (3–4 dias)
**Prioridade**: Alta
**Depende de**: HIST-2026-032 (passageiro agora tem CPF e `PENDING_VERIFICATION`)

---

## 1. Contexto & Motivação

A revisão de produto de maio/2026 identificou três gaps de UX no fluxo do passageiro que causam atrito na ativação e na primeira corrida:

### Gap 1 — Sem endpoint "me" para passageiro

Mesmo problema documentado para o motorista (HIST-2026-031): o app precisa armazenar localmente o UUID do passageiro retornado no cadastro. Sem `GET /api/v1/passengers/me`, uma reinstalação do app ou limpeza de cache resulta em conta inacessível sem suporte humano.

### Gap 2 — Método de pagamento não configurado no onboarding

O passageiro solicita a primeira corrida, o motorista aceita, a viagem acontece, e só no momento de `POST /api/v1/trips/{id}/complete-with-payment` o app descobre que não há `stripePaymentMethodId` configurado. O passageiro precisa:

1. Sair do fluxo de conclusão.
2. Navegar até "Carteiras/Pagamentos".
3. Adicionar cartão via Stripe SDK (HIST-2026-028).
4. Voltar ao fluxo de conclusão.

Este desvio tem alto índice de abandono e gera corridas concluídas sem pagamento pendente de resolução manual. O onboarding deve capturar o cartão **antes** da primeira corrida.

### Gap 3 — Sem endereços salvos

Não existe entidade de local favorito. Cada corrida obriga o passageiro a digitar a origem do zero, inclusive endereços frequentes como casa e trabalho. Isso aumenta o tempo para solicitar uma corrida e é uma das principais funcionalidades esperadas por usuários vindos de Uber/99.

## 2. Objetivo e Escopo

### 2.1 Endpoint `GET /api/v1/passengers/me`

Mesmo padrão documentado em HIST-2026-031 para motoristas.

**Implementação:**
- Novo endpoint no `PassengerController`.
- Extrai `keycloakUserId` (claim `sub`) de `SecurityContextHolder`.
- `PassengerRepositoryPort` recebe novo método: `Optional<Passenger> findByKeycloakId(String keycloakId)`.
- `JpaPassengerRepositoryAdapter` implementa a query via `SpringDataPassengerRepository.findByKeycloakId(String)`.

> **Pré-requisito de dados**: verificar se `PassengerJpaEntity` possui campo `keycloakId`. Se não, adicionar como `VARCHAR(255)` nullable na migration (obrigatório em novos cadastros após esta história).

**Segurança:** `@PreAuthorize("isAuthenticated()")`.

### 2.2 Método de pagamento no onboarding

O objetivo não é obrigar o passageiro a ter um cartão para criar a conta — isso seria barreira excessiva. O objetivo é criar um **momento natural de captura** logo após a verificação de telefone (HIST-2026-032), quando o passageiro ainda está no fluxo de configuração.

**Abordagem: Stripe SetupIntent para vault do cartão**

Diferente do `PaymentMethod` de uso imediato (`pm_xxxxx`), um `SetupIntent` salva o cartão para uso futuro sem cobrar agora. O Stripe retorna um `setupIntent.paymentMethodId` que pode ser reutilizado em múltiplas corridas.

**Fluxo:**
```
1. App solicita → POST /api/v1/passengers/{id}/payment-methods/setup
   Backend cria SetupIntent no Stripe e retorna client_secret
2. App usa Stripe SDK com o client_secret para capturar dados do cartão nativamente
3. Stripe confirma o SetupIntent, retorna pm_xxxxx ao app
4. App registra → POST /api/v1/passengers/{id}/payment-methods
   Body: { "stripePaymentMethodId": "pm_xxxxx", "label": "Visa terminado em 4242" }
5. Backend salva referência (não os dados do cartão) na entidade PassengerPaymentMethod
```

**Nova entidade de domínio `PassengerPaymentMethod`:**
```java
// domain/entities/PassengerPaymentMethod.java
public class PassengerPaymentMethod {
    private final UUID id;
    private final UUID passengerId;
    private final TenantId tenantId;
    private final String stripePaymentMethodId;  // pm_xxxxx — token opaco
    private final String label;                  // "Visa terminado em 4242"
    private boolean isDefault;
    private final LocalDateTime createdAt;
    private LocalDateTime updatedAt;

    public void setAsDefault() { this.isDefault = true; }
    public void unsetDefault() { this.isDefault = false; }
}
```

**Output port `PassengerPaymentMethodRepositoryPort`:**
```java
PassengerPaymentMethod save(PassengerPaymentMethod method);
List<PassengerPaymentMethod> findByPassengerId(UUID passengerId);
Optional<PassengerPaymentMethod> findDefaultByPassengerId(UUID passengerId);
void deleteById(UUID id);
```

**Output port `PaymentSetupPort`** (estende o contexto do `PaymentGatewayPort` existente ou novo port dedicado):
```java
// application/ports/output/PaymentSetupPort.java
public interface PaymentSetupPort {
    String createSetupIntent(String customerId);   // retorna client_secret
    // customerId = stripeCustomerId do passageiro (ver nota abaixo)
}
```

> **Nota — Stripe Customer**: para vault de cartão, o Stripe exige um `Customer` associado. Ao criar o passageiro, o backend deve criar um `Stripe Customer` e armazenar o `stripeCustomerId` no `Passenger`. Adicionar campo `stripeCustomerId` (nullable) à entidade e à migration.

**Novos use cases:**
- `CreatePaymentSetupIntentUseCase` — cria Stripe Customer se não existir, cria SetupIntent, retorna `client_secret`.
- `SavePassengerPaymentMethodUseCase` — recebe `stripePaymentMethodId`, salva `PassengerPaymentMethod`, marca como default se for o primeiro.
- `ListPassengerPaymentMethodsUseCase` — lista métodos salvos do passageiro autenticado.
- `DeletePassengerPaymentMethodUseCase` — remove método salvo.

**Novos endpoints no `PassengerController`:**

| Método | Path | Descrição |
|---|---|---|
| `POST` | `/api/v1/passengers/{id}/payment-methods/setup` | Inicia SetupIntent → retorna client_secret |
| `POST` | `/api/v1/passengers/{id}/payment-methods` | Salva método de pagamento após confirmação Stripe |
| `GET` | `/api/v1/passengers/{id}/payment-methods` | Lista métodos salvos |
| `DELETE` | `/api/v1/passengers/{id}/payment-methods/{methodId}` | Remove método salvo |

**Integração com o fluxo de corrida:**
- `POST /api/v1/trips/{id}/complete-with-payment` passa a aceitar, como alternativa a `stripePaymentMethodId` no body, o campo `useDefaultPaymentMethod: true` — o backend busca o método default do passageiro e aplica.
- Se o passageiro não tiver método salvo e não enviar `stripePaymentMethodId`, retorna 422 com mensagem orientando a configurar um cartão.

### 2.3 Endereços salvos (FavoriteLocation)

**Nova entidade de domínio `FavoriteLocation`:**
```java
// domain/entities/FavoriteLocation.java
public class FavoriteLocation {
    private final UUID id;
    private final UUID passengerId;
    private final TenantId tenantId;
    private final String label;         // "Casa", "Trabalho", "Academia"
    private final Location coordinates; // value object existente (lat/lon)
    private final String address;       // endereço legível por humano

    private LocalDateTime updatedAt;

```

**Output port `FavoriteLocationRepositoryPort`:**
```java
FavoriteLocation save(FavoriteLocation location);
List<FavoriteLocation> findByPassengerId(UUID passengerId);
Optional<FavoriteLocation> findById(UUID id);
void deleteById(UUID id);
```

**Novos use cases:**
- `SaveFavoriteLocationUseCase`
- `ListFavoriteLocationsUseCase`
- `DeleteFavoriteLocationUseCase`

**Novos endpoints** — controller próprio `FavoriteLocationController` em `presentation/controllers/`:

| Método | Path | Descrição |
|---|---|---|
| `POST` | `/api/v1/passengers/{passengerId}/locations` | Salva endereço favorito |
| `GET` | `/api/v1/passengers/{passengerId}/locations` | Lista endereços favoritos |
| `DELETE` | `/api/v1/passengers/{passengerId}/locations/{locationId}` | Remove endereço favorito |

**Limite:** máximo 10 endereços salvos por passageiro (retorna 422 se excedido) — evita acúmulo ilimitado.

## 4. Critérios de Aceite

### Endpoint `me`
- [ ] `GET /api/v1/passengers/me` retorna perfil do passageiro autenticado via `sub` do JWT.
- [ ] Retorna 404 se `keycloakId` não corresponder a nenhum passageiro.
- [ ] Retorna 401 sem token válido.

### Método de Pagamento
- [ ] `POST /passengers/{id}/payment-methods/setup` retorna `client_secret` válido do Stripe.
- [ ] Cria Stripe Customer se `stripeCustomerId` ainda for nulo no passageiro.
- [ ] `POST /passengers/{id}/payment-methods` com `stripePaymentMethodId` válido salva o método e o marca como default se for o primeiro.
- [ ] `GET /passengers/{id}/payment-methods` lista os métodos salvos (somente do passageiro autenticado).
- [ ] `DELETE /passengers/{id}/payment-methods/{methodId}` remove o método; 404 se não encontrado.
- [ ] `POST /trips/{id}/complete-with-payment` com `useDefaultPaymentMethod: true` usa o cartão default do passageiro.
- [ ] `POST /trips/{id}/complete-with-payment` sem cartão configurado e sem `stripePaymentMethodId` retorna 422.

### Endereços Salvos
- [ ] `POST /passengers/{id}/locations` salva endereço e retorna 201 com `FavoriteLocationResponse`.
- [ ] `GET /passengers/{id}/locations` retorna lista de até 10 endereços do passageiro.
- [ ] `POST /passengers/{id}/locations` com mais de 10 endereços existentes retorna 422.
- [ ] `DELETE /passengers/{id}/locations/{locationId}` remove endereço; 403 se o passageiro tentar deletar endereço de outro usuário.
- [ ] Isolamento de tenant: passageiro de tenant A não vê endereços de passageiro de tenant B.

### Geral
- [ ] `mvn test` verde — ArchUnit (24 regras) + testes unitários de todos os novos use cases.
- [ ] `mvn verify` verde — ITs cobrendo `me`, setup intent (mock Stripe), save/list/delete de payment methods e locations.
