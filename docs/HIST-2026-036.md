# HIST-2026-036: Keycloak — Qualidade e Robustez (Senha na Resposta, URL Parsing, Testes de Segurança)

**Data**: 05/05/2026
**Status**: 📋 Planejado
**Tipo**: Quality / Security
**Tamanho**: Pequeno (1–2 dias)
**Prioridade**: Média
**Depende de**: HIST-2026-035 (hardening alto — altera `KeycloakProvisioningAdapter`)

---

## 1. Contexto & Motivação

A revisão da integração Keycloak de maio/2026 identificou três gaps de média prioridade que comprometem a qualidade operacional e a cobertura de testes de segurança.

### Gap 1 — Senha temporária retornada no body da API

`OnboardNewTenantUseCase` gera uma `tempPassword` e a inclui no `OnboardingResult`:

```java
return new OnboardingResult(
    tenantId,
    adminEmail,
    keycloakAdminId,
    tempPassword,          // ← exposta na resposta HTTP
    "Tenant onboarded successfully. ..."
);
```

A senha trafega do backend para o chamador da API via HTTP e aparece em:
- Body da resposta JSON (logado por reverse proxies, APMs, Insomnia/Postman).
- Logs do `SuperAdminController` se houver interceptação de request/response.
- Potencialmente em ferramentas de observabilidade (Zipkin traces, Prometheus labels).

Após HIST-2026-034, o fluxo será diferente: `createDriverAccount` e `createPassengerAccount` não usam senha inicial — enviam e-mail de definição de senha via Keycloak. O `createTenantAdmin` ainda usa `tempPassword`, mas ela deve ir apenas por e-mail, nunca na resposta.

### Gap 2 — `extractIdFromLocation` é frágil

```java
private String extractIdFromLocation(String locationPath) {
    String[] parts = locationPath.split("/");
    return parts[parts.length - 1];  // ← assume estrutura da URL do Keycloak
}
```

A biblioteca `keycloak-admin-client` já oferece `CreatedResponseUtil.getCreatedId(response)` que extrai o ID de forma robusta. Usar `split("/")` sobre a URL é frágil: qualquer trailing slash, query string ou mudança na estrutura de URL do Keycloak quebra silenciosamente (retorna string inválida em vez de lançar exceção).

### Gap 3 — Testes nunca validam a configuração de segurança real

`TestSecurityConfiguration` e `NoSecurityConfiguration` usam `permitAll()` para todos os requests. Isso significa:

- Nenhum teste verifica que `ROLE_DRIVER` é exigido para `/api/v1/drivers/**`.
- Nenhum teste verifica que um passageiro não consegue chamar endpoints de motorista.
- Nenhum teste verifica que o `TenantResolverFilter` bloqueia requests sem `X-Tenant-ID`.
- A configuração de segurança pode estar errada e os testes passarão mesmo assim.

## 2. Objetivo e Escopo

### 2.1 Remover senha temporária da resposta da API

**Alterar `OnboardingResult`:**
```java
// ANTES
public record OnboardingResult(
    UUID tenantId,
    String adminEmail,
    String keycloakAdminId,
    String tempPassword,    // ← remover
    String message
) {}

// DEPOIS
public record OnboardingResult(
    UUID tenantId,
    String adminEmail,
    String keycloakAdminId,
    String message
) {}
```

**Alterar `OnboardNewTenantUseCase`:**
- Remover `tempPassword` do `OnboardingResult`.
- A senha temporária já é enviada por e-mail via `executeActionsEmail` do Keycloak (definida em HIST-2026-034 como link de definição de senha). Para o `createTenantAdmin`, se ainda usar senha temporária, garantir que seja enviada apenas via e-mail Keycloak e nunca retornada pelo backend.

**Alterar `SuperAdminController`:**
- Remover qualquer referência a `tempPassword` no mapeamento da resposta.
- Atualizar `TenantOnboardingResponse` DTO na camada de apresentação — remover campo `tempPassword`.

**Segurança adicional:**
- Garantir que `OnboardNewTenantUseCase` não loga a `tempPassword` em nenhum nível (verificar todos os `log.info/debug` no use case e no adapter).

### 2.2 Substituir `extractIdFromLocation` por `CreatedResponseUtil`

**Alterar `KeycloakProvisioningAdapter`:**

```java
// ANTES
private String extractIdFromLocation(String locationPath) {
    String[] parts = locationPath.split("/");
    return parts[parts.length - 1];
}

// Uso
String userId = extractIdFromLocation(response.getLocation().getPath());

// DEPOIS — usar utilitário da biblioteca
import org.keycloak.admin.client.CreatedResponseUtil;

String userId = CreatedResponseUtil.getCreatedId(response);
```

`CreatedResponseUtil.getCreatedId(response)` trata corretamente:
- Trailing slashes.
- Query strings na URL.
- Respostas com status ≠ 201 (lança `WebApplicationException`).

Remover o método privado `extractIdFromLocation` após a substituição — não há mais uso.

### 2.3 Adicionar testes de segurança com JWT real (MockMvc + Spring Security Test)

Criar suite de testes `SecurityIntegrationTest` em `src/test/.../e2e/` ou `src/test/.../security/` usando `@SpringBootTest` + `MockMvc` + `@WithMockUser` ou JWTs mockados.

**Cenários obrigatórios:**

```
Autenticação:
  ✗ Request sem token → 401
  ✗ Request com token expirado → 401
  ✗ Request com token de role errada → 403

Isolamento de Role:
  ✗ ROLE_PASSENGER tentando GET /api/v1/drivers/{id} → 403
  ✗ ROLE_DRIVER tentando GET /api/v1/passengers/{id} → 403
  ✓ ROLE_DRIVER acessando GET /api/v1/drivers/{id} → 200 (ou 404 se não existir)
  ✓ ROLE_ADMIN acessando GET /api/v1/admin/** → 200

Tenant Filter:
  ✗ Request sem X-Tenant-ID → 400
  ✗ Request com X-Tenant-ID inválido (não UUID) → 400
  ✗ Request com X-Tenant-ID de tenant inexistente → 404
  ✗ JWT com tenant_id diferente do X-Tenant-ID header → 403
  ✗ JWT sem claim tenant_id → 403 (após HIST-2026-035)

Endpoints públicos:
  ✓ GET /actuator/health sem token → 200
  ✓ GET /swagger-ui/** sem token → 200 (se docsPublic=true)
```

**Abordagem técnica:**

Criar `MockJwtFactory` em `src/test/.../security/` que gera JWTs assinados com chave RSA de teste:

```java
public class MockJwtFactory {
    public static String driverToken(UUID tenantId) { ... }
    public static String passengerToken(UUID tenantId) { ... }
    public static String adminToken(UUID tenantId) { ... }
    public static String tokenWithoutTenantClaim() { ... }
    public static String expiredToken(UUID tenantId) { ... }
}
```

Configurar `application-securitytest.properties` com `jwk-set-uri` apontando para chave pública de teste (JWK inline, sem Keycloak real).

**Profile de teste de segurança:**
- Criar profile `securitytest` com `SecurityConfiguration` ativo (não o `TestSecurityConfiguration`).
- `TenantResolverFilter` ativo neste profile.
- `TenantRepositoryPort` mockado para retornar `existsById = true` para tenants de teste.

## 4. Critérios de Aceite

### Senha na Resposta
- [ ] `POST /api/admin/tenants/onboard` não retorna campo de senha na resposta JSON.
- [ ] Grep em todos os logs de `OnboardNewTenantUseCase` e `KeycloakProvisioningAdapter` não encontra `tempPassword` sendo logada.
- [ ] `OnboardingResult` não tem campo de senha — verificável em `OnboardNewTenantUseCaseTest`.

### URL Parsing
- [ ] `KeycloakProvisioningAdapter` não contém método `extractIdFromLocation`.
- [ ] `CreatedResponseUtil.getCreatedId(response)` é usado em todos os pontos onde a resposta de criação Keycloak é processada.
- [ ] Teste unitário de `KeycloakProvisioningAdapter` com resposta mockada de criação valida que o ID é extraído corretamente.

### Testes de Segurança
- [ ] `SecurityIntegrationTest` contém pelo menos os 12 cenários listados no escopo.
- [ ] `mvn test -Dtest=SecurityIntegrationTest` passa verde com `SecurityConfiguration` real ativo.
- [ ] Todos os cenários de 401/403/400 por violação de segurança estão cobertos.
- [ ] Cenários de acesso legítimo (200/404) estão cobertos para cada role.

### Geral
- [ ] `mvn test` verde — ArchUnit + todos os testes unitários existentes inalterados.
- [ ] `mvn verify` verde — ITs existentes não impactados.
