# HIST-2026-038: Push Notifications via Firebase Admin SDK (FCM)

**Data**: 05/05/2026
**Status**: 📋 Planejado
**Tipo**: Infrastructure / Feature
**Tamanho**: Pequeno (1–2 dias)
**Prioridade**: Alta
**Depende de**: HIST-2026-037 (auto-dispatch consome `NotificationServicePort`)

## 1. Contexto & Motivação

O HIST-2026-037 (Motor de Auto-Dispatch) usa `NotificationServicePort` para notificar motoristas sobre corridas disponíveis. O adapter atual é `StubNotificationServiceAdapter` — apenas loga no console, nunca envia nada.

Para o MVP, o RappiDrive opera com um **único aplicativo mobile** (um app de motorista, um app de passageiro) e um **único projeto Firebase**. FCM entrega notificações para Android nativamente e para iOS através da ponte APNs configurada nas settings do projeto Firebase (sem código adicional no backend).

A solução é direta: Firebase Admin SDK no backend, tokens FCM armazenados no banco, `FcmNotificationAdapter` substitui o stub.

```
    │

    │

    ▼
FCM ──► Android (direto)
    └──► iOS (via APNs bridge do Firebase project)
```

### 3.3 Use cases

**`RegisterDeviceTokenUseCase`** em `application/usecases/notification/`:

```
1. Extrair userId e tenantId do contexto (JWT via CurrentUserPort)
2. Verificar se já existe DeviceToken para userId + platform
   → Se existe: atualizar fcmToken (updateToken) e salvar — UPSERT
   → Se não existe: criar novo DeviceToken e salvar
3. Retornar DeviceToken salvo
```

Idempotente: re-registro com mesmo token é no-op; re-registro com novo token atualiza (OS renova token FCM periodicamente).

**`RemoveDeviceTokenUseCase`** em `application/usecases/notification/`:

```
1. Extrair userId e tenantId do contexto
2. DeviceTokenRepositoryPort.deleteByUserIdAndPlatform(userId, tenantId, platform)
```

Chamado no logout do app.

### 3.4 `FcmNotificationAdapter` — implementação de `NotificationServicePort`

```java
// infrastructure/adapters/FcmNotificationAdapter.java
@Component
@Primary
@ConditionalOnProperty(name = "rappidrive.notifications.provider", havingValue = "fcm")
@RequiredArgsConstructor
@Slf4j
public class FcmNotificationAdapter implements NotificationServicePort {

    private final DeviceTokenRepositoryPort deviceTokenRepository;

    @Override
    public boolean sendPushNotification(Notification notification) {
        List<DeviceToken> tokens = deviceTokenRepository.findByUserId(
            notification.getUserId(), notification.getTenantId());

        if (tokens.isEmpty()) {
            log.warn("No FCM tokens for userId={}", notification.getUserId().getValue());
            return false;
        }

        boolean anySuccess = false;
        for (DeviceToken token : tokens) {
            try {
                Message message = Message.builder()
                    .setToken(token.getFcmToken())
                    .setNotification(
                        com.google.firebase.messaging.Notification.builder()
                            .setTitle(notification.getContent().getTitle())
                            .setBody(notification.getContent().getMessage())
                            .build())
                    .putAllData(notification.getContent().getData())
                    .build();

                FirebaseMessaging.getInstance().send(message);
                anySuccess = true;
                log.debug("FCM sent to userId={} platform={}", 
                    notification.getUserId().getValue(), token.getPlatform());

            } catch (FirebaseMessagingException e) {
                if (e.getMessagingErrorCode() == MessagingErrorCode.UNREGISTERED) {
                    // Token inválido — remover para não tentar novamente
                    deviceTokenRepository.deleteByUserIdAndPlatform(
                        token.getUserId(), token.getTenantId(), token.getPlatform());
                    log.info("Removed stale FCM token for userId={}", token.getUserId().getValue());
                } else {
                    log.error("FCM send failed for userId={}: {}", 
                        notification.getUserId().getValue(), e.getMessage());
                }
            }
        }
        return anySuccess;
    }

    @Override
    public boolean isAvailable() {
        try {
            FirebaseMessaging.getInstance();
            return true;
        } catch (Exception e) {
            return false;
        }
    }
}
```

### 3.5 Inicialização do Firebase Admin SDK

```java
// infrastructure/config/FirebaseConfig.java
@Configuration
@ConditionalOnProperty(name = "rappidrive.notifications.provider", havingValue = "fcm")
public class FirebaseConfig {

    @Value("${firebase.credentials.path}")
    private String credentialsPath;

    @Bean
    public FirebaseApp firebaseApp() throws IOException {
        if (!FirebaseApp.getApps().isEmpty()) {
            return FirebaseApp.getInstance();
        }
        GoogleCredentials credentials = GoogleCredentials.fromStream(
            new FileInputStream(credentialsPath));
        FirebaseOptions options = FirebaseOptions.builder()
            .setCredentials(credentials)
            .build();
        return FirebaseApp.initializeApp(options);
    }
}
```

## 4. Infrastructure

### 4.1 Flyway migration

**`V{next}__create_device_tokens.sql`:**
```sql
CREATE TABLE device_tokens (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL,
    tenant_id   UUID NOT NULL,
    fcm_token   TEXT NOT NULL,
    platform    VARCHAR(10) NOT NULL CHECK (platform IN ('ANDROID', 'IOS')),
    created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    UNIQUE (user_id, platform)
);

CREATE INDEX idx_device_tokens_user_id ON device_tokens(user_id);
```

`UNIQUE (user_id, platform)` — garante no máximo um token por usuário por plataforma. O UPSERT no use case resolve conflito atualizando o token existente.

## 5. Critérios de Aceite

### Registro de token
- [ ] `POST /api/v1/notifications/device-token` salva `(userId, tenantId, fcmToken, platform)`.
- [ ] Re-registrar mesmo `userId + platform` com token diferente atualiza o registro (UPSERT).
- [ ] Re-registrar com mesmo token é no-op (updated_at atualiza, sem duplicata).
- [ ] `DELETE /api/v1/notifications/device-token/{platform}` remove o token do banco.

### Envio de notificações
- [ ] `FcmNotificationAdapter.sendPushNotification()` envia via `FirebaseMessaging.getInstance().send()`.
- [ ] Token com erro `UNREGISTERED` é removido automaticamente do banco.
- [ ] Usuário sem tokens registrados: retorna `false`, log de aviso — sem exceção.
- [ ] Usuário com dois devices (Android + iOS) recebe notificação em ambos.

### Integração com Auto-Dispatch (HIST-2026-037)
- [ ] Após `POST /api/v1/trips`, motoristas com FCM token registrado recebem push `TRIP_REQUEST`.
- [ ] Payload data inclui `tripId`, `originLat`, `originLng`, `estimatedFare`.

### Configuração
- [ ] Com `provider=stub` (padrão), `FirebaseApp` não é inicializado — startup sem credenciais Firebase.
- [ ] Com `provider=fcm` e `FIREBASE_CREDENTIALS_PATH` inválido, a aplicação falha na inicialização com mensagem clara.
- [ ] `firebase-service-account.json` está no `.gitignore` — sem risco de commit acidental de credenciais.

### Testes
- [ ] `FcmNotificationAdapterTest` — unit test com `FirebaseMessaging` mockado, cobrindo: envio bem-sucedido, token UNREGISTERED (remoção), usuário sem tokens.
- [ ] `RegisterDeviceTokenUseCaseTest` — cobre criação e upsert.
- [ ] `mvn test` verde — sem ArchUnit violations (`FcmNotificationAdapter` em `infrastructure/adapters/`, `FirebaseConfig` em `infrastructure/config/`).
- [ ] `mvn verify` verde — ITs existentes não impactados (provider=stub nos testes).

## 6. Decisões de Design

| Decisão | Escolhida | Alternativa | Motivo |
|---|---|---|---|
| Provider de push | FCM direto via Firebase Admin SDK | AWS SNS | MVP com único app/tenant — SNS adiciona custo e complexidade desnecessários |
| iOS | FCM entrega via APNs bridge (configurado no Firebase project) | APNS separado | Zero código adicional no backend; Firebase gerencia a bridge |
| Token inválido | Remover do banco + log | Retry | Token UNREGISTERED não se resolve com retry |
| Ativação | `@ConditionalOnProperty(provider=fcm)` | Feature flag | Startup sem Firebase em dev/test; opt-in explícito em produção |
| Credenciais | Arquivo JSON montado via secret / variável de ambiente | Hardcoded / banco | Segurança; rotação sem deploy |
| UPSERT de token | Update no use case (check-then-act) | DB ON CONFLICT | Mantém lógica no use case, alinhado com padrão hexagonal |

## 7. Pós-MVP — Quando Houver Múltiplos Clientes

Quando o RappiDrive tiver tenants distintos com apps próprios (bundle IDs diferentes, Firebase projects diferentes), a solução correta é AWS SNS com Platform Applications por tenant — cada tenant cadastra suas credenciais FCM/APNS, o backend usa o ARN do endpoint como identificador de dispositivo.

A migração é isolada: substituir `FcmNotificationAdapter` por `SnsNotificationAdapter`, alterar `DeviceToken` para armazenar `snsEndpointArn` em vez de `fcmToken`, e adicionar `TenantNotificationConfig`. O port `NotificationServicePort` e o use case `RegisterDeviceTokenUseCase` permanecem os mesmos.
