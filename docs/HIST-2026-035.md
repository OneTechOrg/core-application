## 1. Contexto & Motivação

A revisão da integração Keycloak de maio/2026 identificou três gaps de alta prioridade que comprometem a segurança operacional e a confiabilidade do provisionamento de tenants.

### Gap 1 — Admin client usa grant `password` (deprecado)

`KeycloakConfig` autentica o admin client com `username` + `password` via Resource Owner Password Credentials grant:

```java
KeycloakBuilder.builder()
    .serverUrl(serverUrl)
    .realm("master")
    .clientId(clientId)
    .username(username)
    .password(password)  // @Value("${keycloak.admin.password}") — sem default!
    .build();
```

Problemas:
- Grant `password` está deprecado no OAuth 2.1 — Keycloak 25+ desabilita por padrão.
- A senha do admin trafega pela aplicação e pode aparecer em logs de erro, heap dumps ou APM.
- `keycloak.admin.password` não tem valor padrão — se a variável não existir no ambiente de produção, a aplicação não inicia (falha silenciosa na configuração).
- O client `admin-cli` não deveria ter `Direct Access Grants` habilitado em produção.

### Gap 2 — JWT sem claim `tenant_id` bypassa a validação de tenant

`TenantResolverFilter.isJwtTenantConsistent()` retorna `true` quando o claim `tenant_id` está ausente no JWT:

```java
String jwtTenantId = jwtAuth.getToken().getClaimAsString("tenant_id");
if (jwtTenantId == null) {
    return true;  // ← qualquer JWT sem o claim passa
}
```

Um usuário autenticado via Keycloak mas sem o atributo `tenant_id` configurado no seu perfil consegue enviar `X-Tenant-ID` com o UUID de qualquer tenant e acessar os dados desse tenant. Isso invalida parcialmente o isolamento de tenant para usuários com JWTs "vazios".

### Gap 3 — `createTenantAdmin` não é idempotente e não tem circuit breaker

O `OnboardNewTenantUseCase` chama o Keycloak Admin API em sequência (criar user → setar senha → criar grupo → adicionar ao grupo → atribuir role). Se qualquer passo falhar:

- O Keycloak pode ter criado o usuário mas não atribuído a role — estado inválido.
- A operação lança `IdentityProvisioningException` mas não reverte o que já foi feito no Keycloak.
- Uma re-tentativa do onboarding falha com "usuário já existe".
- Não há circuit breaker — se o Keycloak estiver instável, o onboarding falha em cascata sem degradação graciosa.

## 2. Objetivo e Escopo

### 2.1 Migrar admin client para `client_credentials`

**Criar service account no Keycloak:**
- Criar um client dedicado `rappidrive-backend` (confidential, `service accounts enabled = true`).
- Atribuir as roles necessárias ao service account: `realm-management:manage-users`, `realm-management:manage-groups`, `realm-management:view-users`.
- Desabilitar `Direct Access Grants` em todos os clients que não precisam (incluindo `admin-cli`).

**Alterar `KeycloakConfig`:**
```java
// ANTES
KeycloakBuilder.builder()
    .realm("master")
    .clientId("admin-cli")
    .username(username)
    .password(password)
    .build();

// DEPOIS
KeycloakBuilder.builder()
    .realm(realm)              // realm da aplicação, não "master"
    .clientId(serviceClientId)
    .clientSecret(clientSecret)
    .grantType(OAuth2Constants.CLIENT_CREDENTIALS)
    .build();
```

**Novas properties:**
```properties
keycloak.service-client-id=${KEYCLOAK_SERVICE_CLIENT_ID:rappidrive-backend}
keycloak.service-client-secret=${KEYCLOAK_SERVICE_CLIENT_SECRET}  # obrigatório, sem default
```

**Remover properties antigas:**
- `keycloak.admin.client-id`
- `keycloak.admin.username`
- `keycloak.admin.password`

**Documentação de infraestrutura:** criar `docs/KEYCLOAK_SETUP.md` com passo a passo para criar o service account e configurar as roles mínimas necessárias.

### 2.2 Tornar o claim `tenant_id` obrigatório no JWT

**Alterar `TenantResolverFilter.isJwtTenantConsistent()`:**

```java
// ANTES: claim ausente = passa
if (jwtTenantId == null) {
    return true;
}

// DEPOIS: claim ausente em JWT autenticado = rejeita
if (jwtTenantId == null) {
    log.warn("JWT sem claim tenant_id para request: {} {}", method, requestUri);
    sendErrorResponse(response, HttpServletResponse.SC_FORBIDDEN,
        "JWT must contain tenant_id claim");
    return false;
}
```

**Exceções legítimas sem `tenant_id`:**
- Endpoints públicos já são excluídos pelo `isPublicEndpoint()` — não chegam ao check.
- Super Admin endpoints (`/api/admin/tenants/**`) já estão em `PUBLIC_ENDPOINT_PATTERNS` — não verificam tenant.
- Nenhum outro caso legítimo existe para JWT sem `tenant_id` em endpoints protegidos.

**Keycloak — garantir que o claim seja incluído no token:**
- No Keycloak, o atributo `tenant_id` do usuário deve ser mapeado para o JWT via `User Attribute` mapper no client.
- Documentar esse mapeamento em `docs/KEYCLOAK_SETUP.md`.
- Ao criar usuários via `KeycloakProvisioningAdapter`, o atributo `tenant_id` já é setado — mas o mapper no client precisa estar configurado para que apareça no token.

### 2.3 Tornar `createTenantAdmin` idempotente e resiliente

**Estratégia de idempotência — "check-then-act":**

```
1. Verificar se usuário com esse e-mail já existe no Keycloak
   → Se existe e está em estado válido (enabled, com role e grupo): retornar keycloakId existente
   → Se existe mas incompleto: continuar do passo onde parou (compensação)
2. Se não existe: criar
3. Verificar e atribuir role (idempotente — reatribuir role existente não lança erro)
4. Verificar e adicionar ao grupo (idempotente)
5. Se senha não foi setada: enviar e-mail de definição
```

**Implementação no adapter:**
- Extrair cada passo em método privado idempotente (ex: `ensureUserExists`, `ensureRoleAssigned`, `ensureGroupMembership`).
- Cada método verifica o estado atual antes de agir.
- Se qualquer passo falha após o usuário já ter sido criado, lançar `IdentityProvisioningException` com `keycloakId` no contexto para permitir re-tentativa manual.

**Circuit breaker com Resilience4j:**
```java
// infrastructure/adapters/keycloak/KeycloakProvisioningAdapter.java
@CircuitBreaker(name = "keycloak", fallbackMethod = "createTenantAdminFallback")
@Override
public String createTenantAdmin(TenantId tenantId, Email adminEmail, String tempPassword) { ... }

private String createTenantAdminFallback(TenantId tenantId, Email adminEmail, String tempPassword, Exception e) {
    log.error("Keycloak circuit breaker open — tenant admin creation deferred: {}", tenantId, e);
    throw new IdentityProvisioningException("Keycloak unavailable. Retry later.", e);
}
```

**Configuração em `application.properties`:**
```properties
resilience4j.circuitbreaker.instances.keycloak.sliding-window-size=5
resilience4j.circuitbreaker.instances.keycloak.failure-rate-threshold=60
resilience4j.circuitbreaker.instances.keycloak.wait-duration-in-open-state=30s
resilience4j.circuitbreaker.instances.keycloak.permitted-number-of-calls-in-half-open-state=2
```

> Resilience4j já é dependência do projeto (usado para payment gateway em HIST-2026-*). Reusar a infraestrutura existente.

## 4. Critérios de Aceite

### Admin Client
- [ ] `KeycloakConfig` usa `client_credentials` — sem username/password no código ou properties.
- [ ] `keycloak.service-client-secret` sem valor causa falha explícita na inicialização com mensagem clara.
- [ ] Logs de inicialização não contêm credenciais Keycloak.

### Tenant Claim
- [ ] Request com JWT válido mas sem claim `tenant_id` retorna 403 com mensagem `"JWT must contain tenant_id claim"`.
- [ ] Request com JWT cujo `tenant_id` diverge do header `X-Tenant-ID` continua retornando 403 (comportamento existente preservado).
- [ ] Super Admin endpoints (`/api/admin/**`) continuam acessíveis sem `X-Tenant-ID` (não afetados pela mudança).

### Idempotência e Resiliência
- [ ] Re-executar `POST /api/admin/tenants/onboard` com o mesmo e-mail de admin retorna o `keycloakId` existente em vez de falhar com "usuário já existe".
- [ ] Simulação de falha do Keycloak no passo de atribuição de role: re-tentativa completa com sucesso sem criar usuário duplicado.
- [ ] Circuit breaker abre após 3 falhas consecutivas e retorna 503 com mensagem orientada ao operador.
- [ ] Circuit breaker fecha após recuperação do Keycloak (half-open → closed).

### Geral
- [ ] `mvn test` verde — testes unitários de `isJwtTenantConsistent()` cobrindo claim ausente, claim divergente e claim correto.
- [ ] `mvn verify` verde — IT de onboarding com mock do Keycloak Admin API.
