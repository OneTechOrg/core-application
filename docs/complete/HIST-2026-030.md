# HIST-2026-030: Hardening de Segurança — Onboarding de Motoristas

**Data**: 05/05/2026
**Status**: 📋 Planejado
**Tipo**: Security / Bug Fix
**Tamanho**: Médio (2–3 dias)
**Prioridade**: Crítica

---

## 1. Contexto & Motivação

A revisão de produto de maio/2026 identificou dois gaps críticos de segurança no fluxo de onboarding de motoristas:

### Gap 1 — Ativação de motorista sem guarda de autorização

O endpoint `PUT /api/v1/drivers/{id}/activate` ativa um motorista diretamente, sem verificar se o chamador tem permissão administrativa e sem exigir que o motorista tenha passado pelo workflow de aprovação documentado em HIST-2026-012. Na prática:

- Qualquer cliente autenticado (inclusive o próprio motorista) pode se auto-ativar chamando o endpoint com seu próprio ID.
- Um motorista em `PENDING_APPROVAL` que nunca teve documentos analisados pode começar a operar na plataforma.
- O `ActivateDriverUseCase` invoca `driver.activate()` diretamente, sem consultar o `DriverApproval` associado.

### Gap 2 — Submissão de documentos desconectada do cadastro

O fluxo atual exige dois passos independentes no app mobile:
1. `POST /api/v1/drivers` — cria o motorista com status `PENDING_APPROVAL`.
2. `POST /api/v1/admin/approvals` — submete os documentos para revisão.

O passo 2 não é disparado automaticamente. Se o app mobile esquecer de chamar o segundo endpoint (crash, bug de integração, timeout), o motorista fica em `PENDING_APPROVAL` sem nenhuma aprovação pendente no sistema — estado zumbi invisível para os admins.

Adicionalmente, o endpoint `POST /api/v1/admin/approvals` está sob o prefixo `/admin`, mas é chamado pelo app do **motorista** — nomenclatura e segurança inconsistentes.

---

## 2. Objetivo e Escopo

### 2.1 Corrigir o endpoint de ativação

**Remover** `PUT /api/v1/drivers/{id}/activate` do `DriverController` público.

A ativação de um motorista deve ocorrer **exclusivamente** como efeito colateral da aprovação administrativa:
- Quando um admin chama `POST /api/v1/admin/approvals/{id}/approve`, o `ApproveDriverUseCase` já chama `driver.activate()` internamente.
- Não existe cenário legítimo em que um motorista deva ser ativado sem ter passado por aprovação.

Se houver necessidade de re-ativar um motorista previamente `INACTIVE` (ex: voltou de férias), esse caso deve ser tratado por um endpoint **admin-only** dedicado, com `@PreAuthorize("hasAnyRole('SUPER_ADMIN', 'COMPLIANCE_OFFICER')")`.

### 2.2 Acoplar submissão de documentos ao cadastro

**Mover** a lógica de criação do `DriverApproval` para dentro do `CreateDriverUseCase`. Ao criar um motorista, o use case deve automaticamente criar o registro de aprovação em `PENDING` com a lista de documentos fornecida no cadastro.

**Alterações:**

- `CreateDriverInputPort.CreateDriverCommand` passa a incluir `List<String> documentUrls`.
- `CreateDriverUseCase` recebe `DriverApprovalRepositoryPort` como dependência adicional e cria o `DriverApproval` atomicamente na mesma operação (mesma transação via `@Transactional` no adapter ou via composição no use case).
- O endpoint `POST /api/v1/admin/approvals` (submit) continua existindo para re-envio após rejeição (ver HIST-2026-031), mas deixa de ser o passo obrigatório do onboarding inicial.
- `CreateDriverRequest` passa a incluir `documentUrls: List<String>` (campo obrigatório, mínimo 1 URL).

**Rota do endpoint de submissão:**
- Renomear de `POST /api/v1/admin/approvals` para `POST /api/v1/drivers/{driverId}/approvals` para refletir corretamente que é o motorista submetendo seus documentos (não uma ação admin).

## 3. Arquitetura e Modelagem

### 3.1 Domain

Nenhuma alteração nas entidades de domínio. O `Driver` e o `DriverApproval` permanecem intactos.

### 3.2 Application

**`CreateDriverInputPort`**:
```java
record CreateDriverCommand(
    TenantId tenantId,
    String fullName,
    Email email,
    CPF cpf,
    Phone phone,
    DriverLicense driverLicense,
    List<String> documentUrls  // novo campo obrigatório
) {}
```

**`CreateDriverUseCase`** — adicionar dependência:
```java
public CreateDriverUseCase(
    DriverRepositoryPort driverRepository,
    DriverApprovalRepositoryPort approvalRepository  // nova injeção
) { ... }
```

No `execute()`, após salvar o driver:
```java
DriverApproval approval = new DriverApproval(
    UUID.randomUUID(),
    driver.getId(),
    command.tenantId(),
    serializeDocumentUrls(command.documentUrls())
);
approvalRepository.save(approval);
```

**`ActivateDriverInputPort` / `ActivateDriverUseCase`**: manter para uso interno do `ApproveDriverUseCase`, mas **remover** o endpoint HTTP público correspondente. O use case deixa de ser um port input público exposto via controller.

### 3.3 Infrastructure

- `UseCaseConfiguration`: atualizar a declaração do bean `createDriverUseCase` para injetar `DriverApprovalRepositoryPort`.
- Garantir que a criação de driver + approval esteja na mesma transação: anotar o método no adapter com `@Transactional`, ou criar um `ApplicationService` que componha os dois repositórios sob `@Transactional`.

### 3.4 Presentation

**`CreateDriverRequest`**:
```java
public record CreateDriverRequest(
    UUID tenantId,
    String fullName,
    String email,
    String cpf,
    String phone,
    DriverLicenseDto driverLicense,
    @NotEmpty List<@NotBlank String> documentUrls  // novo campo
) {}
```

**`DriverController`**:
- Remover endpoint `PUT /{id}/activate`.
- Nenhum novo endpoint de ativação pública.

**`ApprovalController`**:
- Remover endpoint `POST /api/v1/admin/approvals` (submit inicial) — coberto agora pelo `CreateDriverUseCase`.
- Manter: `GET /pending`, `POST /{id}/approve`, `POST /{id}/reject`.
- O endpoint de re-envio de documentos (`POST /api/v1/drivers/{driverId}/approvals`) será tratado em HIST-2026-031.

**Segurança**:
- `POST /api/v1/drivers/{id}/approvals/{approvalId}/approve` → `@PreAuthorize("hasAnyRole('SUPER_ADMIN', 'COMPLIANCE_OFFICER')")`
- `POST /api/v1/drivers/{id}/approvals/{approvalId}/reject` → `@PreAuthorize("hasAnyRole('SUPER_ADMIN', 'COMPLIANCE_OFFICER')")`

## 4. Critérios de Aceite

### Funcionais
- [ ] `POST /api/v1/drivers` cria driver + `DriverApproval` em `PENDING` atomicamente. Se a criação do approval falhar, o driver não é persistido (rollback).
- [ ] `POST /api/v1/drivers` retorna 400 se `documentUrls` estiver vazio ou ausente.
- [ ] `PUT /api/v1/drivers/{id}/activate` retorna 404 (endpoint removido). Nenhum cliente legítimo depende deste endpoint — a ativação ocorre via fluxo de aprovação.
- [ ] Um motorista em `PENDING_APPROVAL` que tenta chamar qualquer endpoint de corrida recebe 403.
- [ ] Após `POST /admin/approvals/{id}/approve`, o driver vai para `ACTIVE` — mesmo comportamento de antes, mas agora é o único caminho possível.

### Técnicos & Segurança
- [ ] A criação de driver + approval ocorre em uma única transação. Testado com `@Transactional` + rollback forçado no `CreateDriverIT`.
- [ ] `mvn test` verde — ArchUnit passa, `CreateDriverUseCaseTest` atualizado para incluir `documentUrls` no comando.
- [ ] `mvn verify` verde — `CreateDriverIT` verifica que approval é criado no banco junto com o driver.
- [ ] Nenhuma chamada ao `ActivateDriverInputPort` fora do `ApproveDriverUseCase` (verificável via ArchUnit ou `grep`).
