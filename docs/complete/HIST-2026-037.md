## 1. Contexto & Motivação

Atualmente, quando um passageiro solicita uma corrida via `POST /api/v1/trips`, o `TripCreatedEvent` é publicado e então **nada acontece automaticamente**. A corrida fica em status `REQUESTED` indefinidamente até que um motorista, via polling de `GET /api/v1/drivers/available` (que retorna lista vazia) ou por outro meio, descubra a existência da corrida e se auto-atribua chamando `PUT /api/v1/trips/{id}/assign-driver` com seu próprio `driverId`.

Isso não é operável como produto comercial. A plataforma precisa de um motor de despacho que conecte automaticamente a corrida ao motorista mais adequado disponível.

**Infraestrutura já existente que será aproveitada:**

| Componente | Arquivo | Papel no dispatch |
|---|---|---|
| `TripCreatedEvent` | `domain/events/` | Gatilho de entrada do dispatch |
| `DomainEventPublisher` | `domain/events/` | Canal de entrega do evento |
| `FindAvailableDriversUseCase` | `application/usecases/driver/` | Busca geoespacial paralela de motoristas (PostGIS + 4 zonas) |
| `AssignDriverToTripUseCase` | `application/usecases/trip/` | Atribuição com proteção de concorrência (`version` + `TripConcurrencyException`) |
| `NotificationServicePort` | `application/ports/output/` | Push FCM para app do motorista |
| `OutboxEventProcessor` | `infrastructure/messaging/` | Dispatch assíncrono de eventos (1s, batch 50) |
| `ParallelExecutor` | `application/concurrency/` | Execução paralela com virtual threads |
| `CancellationReason.SYSTEM_AUTO_REMEDIATION` | `domain/valueobjects/` | Motivo de cancelamento automático por timeout |
| `DriverAssignmentMetricsPort` | `application/ports/output/` | Métricas de tentativas e latência de atribuição |

## 2. Objetivo e Escopo

### 2.1 Estratégia de Dispatch — MVP Broadcast + First-Come-First-Served

Dado que WebSocket não está implementado (Cluster C, futuro), o dispatch do MVP usa:

1. **Broadcast paralelo**: ao criar uma corrida, notificar via push FCM todos os motoristas disponíveis num raio configurável simultaneamente.
2. **First-come-first-served**: o primeiro motorista que chamar `PUT /api/v1/trips/{id}/assign-driver` vence. Os demais recebem `TripAlreadyAcceptedException` — o app exibe "corrida aceita por outro motorista".
3. **Timeout automático**: se nenhum motorista aceitar dentro do TTL configurado, a corrida é cancelada com `SYSTEM_AUTO_REMEDIATION`.

Esta estratégia:
- Reutiliza toda a infraestrutura existente sem reescritas.
- A proteção de concorrência (`version` + `TripConcurrencyException`) já está implementada no `AssignDriverToTripUseCase`.
- Não requer WebSocket nem fila de mensagens externa.
- É substituível por dispatch sequencial (Uber-style) quando WebSocket for implementado.

### 2.2 Fluxo completo após esta história

```
1. Passageiro → POST /api/v1/trips
   └── CreateTripUseCase
       ├── Cria Trip (status: REQUESTED)
       └── Publica TripCreatedEvent

2. TripCreatedEvent → TripAutoDispatchHandler (async, virtual thread)
   └── AutoDispatchTripUseCase
       ├── FindAvailableDriversUseCase (PostGIS, raio configurável, paralelo)
       ├── Se nenhum motorista: agenda retry em N segundos (até max_retries)
       ├── Se ainda sem motorista após retries: cancela corrida (SYSTEM_AUTO_REMEDIATION)
       └── Para cada motorista encontrado (paralelo):
           └── NotificationServicePort.sendPushNotification(offerNotification)

3. Motorista recebe push → app exibe "Nova corrida disponível!"
   └── GET /api/v1/trips/available  (novo endpoint — lista corridas REQUESTED próximas)
   └── PUT /api/v1/trips/{id}/assign-driver (self-assign)
       ├── Sucesso: corrida muda para DRIVER_ASSIGNED, motorista para BUSY
       └── TripAlreadyAcceptedException: app exibe "corrida aceita por outro motorista"

4. TripAutoDispatchTimeoutJob (scheduled, a cada 30s)
   └── Busca corridas em REQUESTED com requestedAt > TTL configurado
   └── Cancela com SYSTEM_AUTO_REMEDIATION
   └── Publica TripCancelledEvent → notifica passageiro
```

### 2.3 Endpoint `GET /api/v1/trips/available`

Atualmente retorna lista vazia. Deve passar a retornar corridas em `REQUESTED` próximas ao motorista autenticado.

**Query:** corridas em `REQUESTED` do tenant, ordenadas por `requestedAt` ASC (mais antigas primeiro), filtradas por distância entre `trip.origin` e `driver.currentLocation` ≤ raio configurável.

**Input port:** `ListAvailableTripsInputPort` com `ListAvailableTripsCommand(TenantId, Location driverLocation, double radiusKm)`.

**Use case:** `ListAvailableTripsUseCase`.

**Novo método no `TripRepositoryPort`:**
```java
List<Trip> findByStatusAndOriginNear(TenantId tenantId, TripStatus status,
                                     Location driverLocation, double radiusKm);
```

### 3.2 Application

#### Novo use case: `AutoDispatchTripUseCase`

**Localização:** `application/usecases/trip/AutoDispatchTripUseCase.java`

**Input port:** `AutoDispatchTripInputPort` em `application/ports/input/trip/`

```java
public interface AutoDispatchTripInputPort {
    record DispatchCommand(TripId tripId, TenantId tenantId, Location origin, int attemptNumber) {}
    void execute(DispatchCommand command);
}
```

**Dependências do use case:**
- `FindAvailableDriversInputPort` — busca motoristas disponíveis
- `NotificationServicePort` — envia push para motoristas
- `TripRepositoryPort` — verifica estado atual da trip antes de despachar
- `DriverAssignmentMetricsPort` — registra métricas de dispatch
- `DomainEventPublisher` — publica `TripDispatchFailedEvent`

**Lógica:**
```
1. Buscar trip pelo ID — se não existir ou não estiver em REQUESTED: abortar silenciosamente
   (pode ter sido aceita ou cancelada entre o evento e a execução)
2. FindAvailableDriversUseCase com origin da trip como centro, raio configurável
3. Se lista vazia:
   - Se attemptNumber < max_attempts: re-agendar via @Async com delay
   - Se attemptNumber >= max_attempts: publicar TripDispatchFailedEvent
   - Retornar
4. Para cada driver (paralelo via ParallelExecutor):
   - Construir Notification de oferta (tipo: TRIP_OFFER)
   - NotificationServicePort.sendPushNotification(notification)
   - Registrar métrica de notificação enviada
5. Registrar métricas de dispatch (drivers notificados, raio usado, attempt)
```

#### Novo handler: `TripAutoDispatchHandler`

**Localização:** `infrastructure/config/` ou `infrastructure/messaging/` (implementa `DomainEventHandler<TripCreatedEvent>`)

**Responsabilidade:** receber `TripCreatedEvent`, executar `AutoDispatchTripUseCase` de forma assíncrona via `@Async` com virtual thread.

```java
@Component
public class TripAutoDispatchHandler implements DomainEventHandler<TripCreatedEvent> {

    private final AutoDispatchTripInputPort autoDispatchUseCase;

    @Override
    @Async("virtualThreadExecutor")
    public void handle(TripCreatedEvent event) {
        autoDispatchUseCase.execute(new DispatchCommand(
            event.tripId(), event.tenantId(), parseOrigin(event.origin()), 1
        ));
    }

    @Override
    public boolean canHandle(DomainEvent event) {
        return event instanceof TripCreatedEvent;
    }
}
```

> **Registro do handler:** em `UseCaseConfiguration` ou `@PostConstruct`, chamar `DomainEventPublisher.instance().register(tripAutoDispatchHandler)`.
>
> **Cuidado com ThreadLocal:** `DomainEventPublisher` usa `ThreadLocal<DomainEventPublisher>` — o handler `@Async` rodará em uma thread diferente, portanto em um publisher diferente. O registro deve ocorrer na thread do handler ou via um singleton compartilhado. **Recomendação:** converter `DomainEventPublisher` de `ThreadLocal` para singleton Spring Bean (`@Bean @Scope("singleton")`) nesta história, o que também resolve o problema de eventos perdidos em contexts assíncronos.

## 4. Critérios de Aceite

### Auto-Dispatch
- [ ] Após `POST /api/v1/trips`, dentro de 2 segundos, todos os motoristas disponíveis no raio configurado recebem push notification de oferta de corrida.
- [ ] Se nenhum motorista estiver disponível, o dispatch re-tenta até `max_attempts` vezes com `retry_delay_ms` de intervalo.
- [ ] Após esgotar as tentativas sem motorista, a corrida é cancelada com `CancellationReason.SYSTEM_AUTO_REMEDIATION` e o passageiro recebe notificação.
- [ ] `TripDispatchFailedEvent` é publicado via Outbox quando o dispatch falha definitivamente.

### Corridas Disponíveis
- [ ] `GET /api/v1/drivers/available` retorna lista de corridas em `REQUESTED` próximas ao motorista autenticado, ordenadas por `requestedAt` ASC.
- [ ] `GET /api/v1/drivers/available` retorna lista vazia se não há corridas próximas — sem erro.
- [ ] `TripOfferResponse` inclui `distanceFromDriverKm` para o app exibir a distância até a origem.

### Timeout Automático
- [ ] Corridas em `REQUESTED` há mais de `trip_ttl_minutes` são canceladas automaticamente pelo `TripAutoDispatchTimeoutJob`.
- [ ] O job roda a cada `timeout_check_rate_ms` e cancela em lote (não uma a uma).
- [ ] Passageiro recebe notificação de cancelamento por "sem motoristas disponíveis".

### Concorrência
- [ ] Dois motoristas tentando aceitar a mesma corrida simultaneamente: um recebe 200, o outro recebe 409 (`TripAlreadyAcceptedException`) — sem corrida com dois motoristas.
- [ ] Motorista que recebe 409 recebe mensagem clara no app: "corrida já aceita por outro motorista".

### `DomainEventPublisher` como Bean Singleton
- [ ] Handlers registrados em threads diferentes do publisher recebem eventos corretamente.
- [ ] `TripAutoDispatchHandler` recebe `TripCreatedEvent` mesmo sendo `@Async`.
- [ ] Retrocompatibilidade: `DomainEventPublisher.instance()` ainda funciona (via delegação ao bean singleton).

### Geral
- [ ] `mvn test` verde — ArchUnit + testes unitários de `AutoDispatchTripUseCase` (mock de todos os ports).
- [ ] `mvn verify` verde — IT de ponta-a-ponta: cria trip → verifica driver notificado (mock FCM) → driver aceita → status DRIVER_ASSIGNED.
- [ ] Métricas de dispatch registradas em `DriverAssignmentMetricsPort` (drivers notificados, raio, tentativas, latência).

## 5. Decisões de Design e Trade-offs

| Decisão | Escolhida | Alternativa | Motivo |
|---|---|---|---|
| Estratégia de oferta | Broadcast para todos os motoristas próximos | Oferta sequencial (Uber-style) | Sequencial requer WebSocket para timeout por motorista. Broadcast aproveita concorrência existente. |
| Trigger do dispatch | `DomainEventHandler<TripCreatedEvent>` | Job periódico de varredura | Handler é mais imediato. Job como fallback de segurança. |
| `DomainEventPublisher` | Converter para Bean Singleton | Manter ThreadLocal | ThreadLocal quebra em contextos async. Singleton é o padrão correto para Spring. |
| Re-tentativas | Delay fixo configurável | Backoff exponencial | Backoff é overkill para 5 min de TTL. Delay fixo é mais simples de monitorar. |
| Cleanup de corridas expiradas | `@Scheduled` job | Trigger por evento | Job é mais confiável — funciona mesmo se o dispatch inicial falhar silenciosamente. |

## 6. Limitações do MVP e Evolução Futura

- **Sem priorização de motoristas**: todos os disponíveis no raio recebem oferta simultaneamente. Futuro: priorizar por rating, distância e taxa de aceitação (Cluster C).
- **Sem oferta exclusiva**: o motorista não tem garantia de que a corrida ainda está disponível quando abre o app. Futuro: WebSocket com offer exclusivo + timeout por motorista.
- **Raio fixo**: `search-radius-km` é global. Futuro: raio dinâmico por demanda (expandir se poucos motoristas disponíveis).
- **Push como único canal**: se o FCM falhar ou o app estiver fechado, o motorista perde a notificação. Futuro: fallback para SMS ou polling melhorado.
