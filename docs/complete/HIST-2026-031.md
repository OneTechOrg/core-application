# HIST-2026-031: UX Flow — Gaps de Produto no Onboarding de Motoristas

**Data**: 05/05/2026
**Status**: 📋 Planejado
**Tipo**: Feature / UX
**Tamanho**: Médio (2–3 dias)
**Prioridade**: Alta
**Depende de**: HIST-2026-030 (deve estar deployado antes — altera `CreateDriverUseCase`)

---

## 1. Contexto & Motivação

A revisão de produto de maio/2026 identificou três gaps de UX/produto no fluxo de onboarding de motoristas que causam atrito operacional:

### Gap 1 — Sem endpoint "me" para motorista

O app mobile precisa do ID do motorista autenticado para fazer qualquer operação subsequente (buscar perfil, atualizar localização, aceitar corridas). Hoje, o app deve armazenar localmente o UUID retornado no `POST /drivers` no momento do cadastro. Se o app for reinstalado, o usuário perder o cache, ou o token expirar, não há forma de recuperar o ID a partir do JWT sem chamar um endpoint dedicado.

`GET /api/v1/drivers/me` está documentado como `🔜 Não implementado` na `MOBILE_API_DOCUMENTATION.md`.

### Gap 2 — Sem endpoint "ficar offline"

O motorista não tem como sinalizar explicitamente que está encerrando o expediente. O fluxo atual assume que o app simplesmente para de enviar localização, mas:

- O status permanece `ACTIVE` indefinidamente se o app travar ou perder conexão.
- Motoristas offline mas com status `ACTIVE` aparecem na busca de `FindAvailableDriversUseCase`, distorcendo a oferta percebida pelo sistema de despacho.
- Não há como o painel admin distinguir "motorista parado" de "motorista com app morto".

Um mecanismo de timeout/desativação automática foi planejado mas nunca implementado.

### Gap 3 — Rejeição é estado terminal

O workflow de aprovação (HIST-2026-012) implementa `PENDING → APPROVED` e `PENDING → REJECTED`, mas não prevê o que acontece depois de uma rejeição:

- O motorista recebe o motivo da rejeição, mas não tem como corrigir os documentos e reenviar.
- O admin não tem como re-abrir a análise sem intervenção direta no banco de dados.
- Na prática, um motorista rejeitado por erro de upload de CNH deve criar um novo cadastro com outro e-mail — o que gera duplicatas e fraude de dados.

## 2. Objetivo e Escopo

### 2.1 Endpoint `GET /api/v1/drivers/me`

Retorna o perfil do motorista autenticado a partir do `sub` claim do JWT.

**Implementação:**
- Novo endpoint no `DriverController`.
- Extrai o `keycloakUserId` (claim `sub`) de `SecurityContextHolder`.
- O `DriverRepositoryPort` (output port) recebe novo método: `Optional<Driver> findByKeycloakId(String keycloakId)`.
- O `JpaDriverRepositoryAdapter` implementa a query via `SpringDataDriverRepository.findByKeycloakId(String)`.

> **Pré-requisito de dados**: o campo `keycloakId` deve existir na entidade `Driver` (domínio + JPA). Se ainda não existe, deve ser adicionado como `String keycloakId` (nullable na migration, obrigatório em novos cadastros). Verificar se `DriverJpaEntity` já possui este campo antes de escrever a migration.

**Input port:** novo `GetDriverByKeycloakIdInputPort` (ou enriquecer o `GetDriverInputPort` existente com método por keycloakId — avaliar qual evita proliferação de ports desnecessária).

**Segurança:** `@PreAuthorize("isAuthenticated()")` — o motorista só pode ver seu próprio perfil (o ID vem do token, não do path).

### 2.2 Endpoint `PATCH /api/v1/drivers/{id}/go-offline`

Coloca o motorista em `INACTIVE` e limpa a localização corrente.

**Implementação:**
- Novo use case `DeactivateDriverUseCase` implementando `DeactivateDriverInputPort`.
- O domínio já possui `driver.deactivate()` — nenhuma alteração no `Driver.java`.
- O endpoint aceita `PATCH` (não `PUT`) pois é uma mudança parcial de estado, não uma substituição completa.
- Segurança: `@PreAuthorize("isAuthenticated()")` + verificação de que o `id` no path corresponde ao motorista do JWT (um motorista não pode desativar outro).

**Timeout automático (complementar, mesma história):**
- Criar um `@Scheduled` job `DriverOfflineTimeoutJob` em `infrastructure/scheduling/` que roda a cada 5 minutos.
- Busca motoristas com status `ACTIVE` cuja `updatedAt` de localização é anterior a 15 minutos.
- Chama `driver.deactivate()` e persiste — com log de aviso.
- Configurável via `application.properties`: `rappidrive.driver.offline-timeout-minutes=15`.

### 2.3 Re-envio de documentos após rejeição

Permitir que um motorista rejeitado submeta novos documentos, gerando um novo `DriverApproval` em `PENDING`.

**Regras de negócio:**
- Só é possível re-enviar se o último `DriverApproval` do motorista estiver em `REJECTED`.
- Se já existe um `PENDING` ativo, re-envio retorna 409 (evita spam de submissions).
- O histórico de approvals anteriores é preservado — não há sobrescrita do registro rejeitado.
- Máximo de re-envios configurável via `application.properties`: `rappidrive.driver.max-resubmissions=3` (retorna 422 se excedido, para forçar análise humana).

**Endpoint:** `POST /api/v1/drivers/{driverId}/approvals`

**Use case:** `ResubmitDriverDocumentsUseCase` implementando `ResubmitDriverDocumentsInputPort`.

**Command:**
```java
record ResubmitDriverDocumentsCommand(
    UUID driverId,
    List<String> documentUrls
) {}
```

**Lógica:**
1. Buscar driver — 404 se não encontrado.
2. Buscar último approval do driver — 409 se status for `PENDING`.
3. 422 se contagem de approvals `REJECTED` ≥ `maxResubmissions`.
4. Criar novo `DriverApproval` em `PENDING` com os novos documentUrls.
5. Publicar evento `DriverDocumentsResubmittedEvent` via Outbox.

**Novo output port method em `DriverApprovalRepositoryPort`:**
```java
Optional<DriverApproval> findLatestByDriverId(UUID driverId);
long countByDriverIdAndStatus(UUID driverId, ApprovalStatus status);
```

- `ResubmitDriverDocumentsInputPort`

Enriquecer `DriverApprovalRepositoryPort` com `findLatestByDriverId` e `countByDriverIdAndStatus`.

### 3.3 Infrastructure

- `JpaDriverRepositoryAdapter`: implementar `findByKeycloakId`.
- `SpringDataDriverRepository`: adicionar `Optional<DriverJpaEntity> findByKeycloakId(String keycloakId)`.
- `JpaDriverApprovalRepositoryAdapter`: implementar `findLatestByDriverId` e `countByDriverIdAndStatus`.
- `DriverOfflineTimeoutJob` em `infrastructure/scheduling/` com `@Scheduled(fixedRateString = "${rappidrive.driver.offline-timeout-rate-ms:300000}")`.

## 4. Critérios de Aceite

### Endpoint `me`
- [ ] `GET /api/v1/drivers/me` retorna o perfil do motorista autenticado com base no `sub` do JWT.
- [ ] Retorna 404 se o `keycloakId` do token não corresponde a nenhum motorista cadastrado.
- [ ] Retorna 401 se a requisição não tiver token válido.

### Go-Offline
- [ ] `PATCH /api/v1/drivers/{id}/go-offline` muda status para `INACTIVE` e limpa `currentLocation`.
- [ ] Retorna 403 se o motorista tentar desativar outro motorista (ID no path ≠ sub do JWT).
- [ ] Retorna 409 se o motorista já estiver `INACTIVE` ou `BLOCKED`.
- [ ] Motorista `INACTIVE` não aparece em `GET /api/v1/drivers/nearby`.
- [ ] `DriverOfflineTimeoutJob` desativa motoristas `ACTIVE` sem update de localização por mais de 15 minutos (testável com `@SpringBootTest` + clock mockado).

### Re-envio de Documentos
- [ ] `POST /api/v1/drivers/{id}/approvals` com `documentUrls` válido cria novo `DriverApproval` em `PENDING` quando o último approval está em `REJECTED`.
- [ ] Retorna 409 se já existe um approval em `PENDING` para o mesmo motorista.
- [ ] Retorna 422 se o motorista excedeu o limite de re-envios (`max-resubmissions`).
- [ ] O approval anterior (`REJECTED`) é preservado no banco — não é sobrescrito.
- [ ] Evento `DriverDocumentsResubmittedEvent` é publicado via Outbox e aparece na tabela `outbox_event`.

### Geral
- [ ] `mvn test` verde — ArchUnit (24 regras) + testes unitários de todos os novos use cases.
- [ ] `mvn verify` verde — ITs cobrindo `me`, go-offline e re-envio com Testcontainers.
