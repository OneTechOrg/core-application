# HIST-2026-040: Vinculação Keycloak ao Perfil de Motorista e Passageiro (MVP sem Admin API)

**Data**: 05/05/2026
**Status**: 📋 Planejado
**Tipo**: Feature / Security
**Tamanho**: Pequeno (1–2 dias)
**Prioridade**: Crítica — bloqueador de desenvolvimento
**Depende de**: nada (pré-requisito para HIST-2026-031, HIST-2026-033)
**Substitui**: HIST-2026-034 (provisionamento via Admin API — adiado para pós-MVP)

---

## 1. Contexto & Motivação

### Bloqueador de startup

`KeycloakConfig` injeta `keycloak.admin.password` sem valor padrão:

```java
@Value("${keycloak.admin.password}")  // ← sem default → BeanCreationException na inicialização
private String password;
```

A aplicação não sobe em nenhum ambiente onde `KEYCLOAK_ADMIN_PASSWORD` não está definido. Isso bloqueia o desenvolvimento local e qualquer pipeline de CI que não configure essa variável.

### Gap de identidade: perfil sem keycloakId

`Driver` e `Passenger` não têm campo `keycloakId`. Quando um motorista autentica via Keycloak e chama `GET /api/v1/drivers/me`, não existe forma de localizar o perfil pelo JWT `sub`. Não há endpoint `/me` implementado.

`CreateDriverUseCase` e `CreatePassengerUseCase` nunca leem o JWT — criam perfis sem vínculo com a identidade Keycloak.

### Por que Keycloak Admin não é necessário para MVP

O fluxo correto para MVP com tenant único:

```
Mobile app → Keycloak self-registration → login → JWT (sub + ROLE_DRIVER)
     ↓
POST /api/v1/drivers (com JWT)
     ↓
CreateDriverUseCase extrai sub do JWT via CurrentUserPort
     ↓
Driver.keycloakId = JWT.sub → salvo no banco
     ↓
GET /api/v1/drivers/me → lookup por keycloakId
```

Roles (`ROLE_DRIVER`, `ROLE_PASSENGER`) são configuradas uma vez no Keycloak como default roles do client — sem Admin API, sem código.

As histórias HIST-2026-034 (Admin API para provisioning), HIST-2026-035 (client_credentials) e HIST-2026-036 (qualidade Keycloak) são **pós-MVP**.

## 2. Objetivo e Escopo

### 2.1 Remover bloqueador de startup — tornar Keycloak Admin opcional

**Opção A — Dar default vazio à propriedade:**
```properties
# application.properties
keycloak.admin.password=${KEYCLOAK_ADMIN_PASSWORD:}
```

**Opção B — `@ConditionalOnProperty` em `KeycloakConfig`:**
```java
@Configuration
@ConditionalOnProperty(name = "keycloak.admin.enabled", havingValue = "true", matchIfMissing = false)
public class KeycloakConfig { ... }
```

Com a opção B, `KeycloakProvisioningAdapter` também precisa ser condicional (ou ter um stub no lugar). **Escolher Opção B** — mais explícito, evita que o admin client tente conectar com senha vazia.

Criar `StubIdentityProvisioningAdapter` anotado com `@ConditionalOnMissingBean(IdentityProvisioningPort.class)` como fallback quando `keycloak.admin.enabled=false`. O stub lança `UnsupportedOperationException` com mensagem clara para operações que exigem o Admin API.

**`application.properties`** — padrão dev:
```properties
keycloak.admin.enabled=false   # padrão: sem Admin API em dev/MVP
# keycloak.admin.enabled=true  # ativar quando tenant onboarding automático for necessário
```

### 2.2 Adicionar `keycloakId` a `Driver` e `Passenger`

**`Driver`** em `domain/entities/Driver.java`:
```java
// Novo campo (nullable até o perfil ser vinculado)
private String keycloakId;

// Novo método de vínculo
public void linkKeycloakAccount(String keycloakId) {
    if (keycloakId == null || keycloakId.isBlank()) {
        throw new IllegalArgumentException("keycloakId não pode ser nulo");
    }
    if (this.keycloakId != null && !this.keycloakId.equals(keycloakId)) {
        throw new IllegalStateException("Driver já está vinculado a outra conta Keycloak");
    }
    this.keycloakId = keycloakId;
}

public String getKeycloakId() { return keycloakId; }
```

**`Passenger`** — mesmo padrão: campo `keycloakId` + método `linkKeycloakAccount(String)`.

### 2.3 `CreateDriverUseCase` — vincular keycloakId na criação

```java
public class CreateDriverUseCase implements CreateDriverInputPort {

    private final DriverRepositoryPort driverRepository;
    private final CurrentUserPort currentUser;

    // ...

    @Override
    public Driver execute(CreateDriverCommand command) {
        if (driverRepository.existsByEmail(command.email())) {
            throw new IllegalArgumentException("Driver with email " + command.email() + " already exists");
        }
        if (driverRepository.existsByCpf(command.cpf())) {
            throw new IllegalArgumentException("Driver with CPF " + command.cpf() + " already exists");
        }

        Driver driver = new Driver(
            UUID.randomUUID(),
            command.tenantId(),
            command.fullName(),
            command.email(),
            command.cpf(),
            command.phone(),
            command.driverLicense()
        );

        // Vincular identidade Keycloak se usuário autenticado
        currentUser.getCurrentUser().ifPresent(user ->
            driver.linkKeycloakAccount(user.userId().toString())
        );

        return driverRepository.save(driver);
    }
}
```

**`CreatePassengerUseCase`** — mesmo padrão.

`UseCaseConfiguration` — passar `CurrentUserPort` nos beans de criação de driver e passenger.

### 2.4 `findByKeycloakId` nos ports de repositório

**`DriverRepositoryPort`** — novo método:
```java
Optional<Driver> findByKeycloakId(String keycloakId, TenantId tenantId);
```

**`PassengerRepositoryPort`** — idem:
```java

```

### 2.5 Use cases de `/me`

**`GetCurrentDriverUseCase`** em `application/usecases/driver/`:
```java
public class GetCurrentDriverUseCase implements GetCurrentDriverInputPort {

    private final DriverRepositoryPort driverRepository;
    private final CurrentUserPort currentUser;

    @Override
    public Driver execute(TenantId tenantId) {
        CurrentUserPort.CurrentUser user = currentUser.getCurrentUser()
            .orElseThrow(() -> new IllegalStateException("No authenticated user"));

        return driverRepository.findByKeycloakId(user.userId().toString(), tenantId)
            .orElseThrow(() -> new DriverNotFoundException(
                "No driver profile linked to the authenticated account"));
    }
}
```

**`GetCurrentPassengerUseCase`** — mesmo padrão.

### 2.6 Endpoints `/me`

**`GET /api/v1/drivers/me`** em `DriverController`:
```java
@GetMapping("/me")
@PreAuthorize("hasRole('DRIVER')")
public ResponseEntity<DriverResponse> getMe() {
    TenantId tenantId = TenantContext.getCurrentTenantId();
    Driver driver = getCurrentDriver.execute(tenantId);
    return ResponseEntity.ok(driverDtoMapper.toResponse(driver));
}
```

**`GET /api/v1/passengers/me`** em `PassengerController` — mesmo padrão.

### 2.7 Flyway migration — coluna `keycloak_id`

**`V{next}__add_keycloak_id_to_drivers_and_passengers.sql`:**
```sql
ALTER TABLE drivers   ADD COLUMN keycloak_id VARCHAR(255) UNIQUE;
ALTER TABLE passengers ADD COLUMN keycloak_id VARCHAR(255) UNIQUE;

CREATE INDEX idx_drivers_keycloak_id   ON drivers(keycloak_id);
CREATE INDEX idx_passengers_keycloak_id ON passengers(keycloak_id);
```

`UNIQUE` — um Keycloak user mapeia para no máximo um perfil de motorista e um de passageiro (por tenant, dado que o `tenant_id` filtra por Hibernate).

### 2.8 Documentação de setup Keycloak para MVP

Criar `docs/KEYCLOAK_SETUP.md` com:

```markdown
## Setup mínimo para MVP (tenant único, sem Admin API)

### 1. Criar realm `rappidrive`
### 2. Criar dois clients: `driver-app` e `passenger-app`
   - Access Type: public (mobile apps não guardam secrets)
   - Valid redirect URIs: URIs do app mobile
### 3. Criar realm roles: `ROLE_DRIVER`, `ROLE_PASSENGER`
### 4. Configurar default roles por client:
   - `driver-app` → Default Client Scope → atribuir `ROLE_DRIVER`
   - `passenger-app` → Default Client Scope → atribuir `ROLE_PASSENGER`
   → Todo usuário que fizer login via driver-app recebe ROLE_DRIVER no JWT automaticamente
### 5. Habilitar self-registration no realm (opcional)
   → Realm Settings → Login → User Registration: ON
### 6. Mapear claim `tenant_id` no token:
   → Clients → driver-app → Mappers → Add mapper → User Attribute
   → Name: tenant_id, User Attribute: tenant_id, Token Claim Name: tenant_id
   → Atribuir atributo `tenant_id` manualmente a cada usuário criado
```

## 4. Critérios de Aceite

### Startup sem Keycloak Admin
- [ ] `mvn spring-boot:run` sem `KEYCLOAK_ADMIN_PASSWORD` definido sobe sem erro.
- [ ] Com `keycloak.admin.enabled=false` (padrão), `KeycloakConfig` e `KeycloakProvisioningAdapter` não são instanciados.
- [ ] `StubIdentityProvisioningAdapter` é o bean ativo quando Admin está desabilitado.
- [ ] `OnboardNewTenantUseCase` com stub lança exceção com mensagem orientada ao operador.

### Criação com keycloakId
- [ ] `POST /api/v1/drivers` com JWT válido cria `Driver` com `keycloakId = JWT.sub`.
- [ ] `POST /api/v1/drivers` sem JWT autenticado cria `Driver` com `keycloakId = null` (não falha — compatibilidade com testes de integração existentes).
- [ ] Tentativa de criar segundo `Driver` com mesmo `keycloakId` falha com 409.

### Endpoints /me
- [ ] `GET /api/v1/drivers/me` com JWT retorna o perfil do motorista vinculado ao `sub`.
- [ ] `GET /api/v1/drivers/me` quando não existe perfil vinculado retorna 404.
- [ ] `GET /api/v1/passengers/me` — mesmo comportamento.

### Testes
- [ ] `CreateDriverUseCaseTest` — cobre: criação com keycloakId (CurrentUserPort mockado), criação sem usuário autenticado.
- [ ] `GetCurrentDriverUseCaseTest` — cobre: perfil encontrado, perfil não encontrado.
- [ ] `mvn test` verde — ArchUnit não violado (`StubIdentityProvisioningAdapter` em `infrastructure/adapters/`).

## 6. Decisões de Design

| Decisão | Escolhida | Alternativa | Motivo |
|---|---|---|---|
| Keycloak Admin para MVP | Desabilitado (`keycloak.admin.enabled=false`) | Manter ativo com password vazio | Conditional explícito evita conexão com senha vazia; não bloqueia startup |
| Criação de conta Keycloak | Self-registration via mobile + JWT | Backend chama Admin API | Sem Admin API no runtime MVP; mobile app é responsável pelo cadastro Keycloak |
| Roles no token | Default Client Scope no Keycloak (configuração manual) | Admin API atribui role após registro | Zero código; configuração única no realm |
| `keycloakId` como String | String (UUID.toString()) | UUID | JWT `sub` é string; evita conversão; mantém compatibilidade com diferentes emissores |
| Criação sem JWT autenticado | `keycloakId = null` (não falha) | Obrigatório | Testes de integração existentes não têm JWT; não quebra ITs |
