## 1. Contexto

A camada de infraestrutura da entidade `ServiceArea` está **100% implementada**: migration Flyway (V18), entidade JPA com `@Filter` de tenant, adapter, Spring Data repository, mapper e output port. O domínio também está completo: `ServiceArea.create()`, `deactivate()`, `activate()`.

O que falta é a **fatia de aplicação e apresentação**: input ports, use cases, DTOs, controller e wiring. Sem esta história, a única forma de criar/gerir áreas de serviço é via `OnboardNewTenantUseCase` (que cria uma área no onboarding, mas não expõe gestão posterior).

A funcionalidade é **crítica para produção**: admins precisam poder adicionar novas zonas de cobertura, desativar áreas temporariamente e consultar as áreas ativas por tenant.

---

## 2. Escopo

### 2.1 Operações da API

| Endpoint | Papel | Descrição |
|---|---|---|
| `GET /api/v1/service-areas` | `AUTHENTICATED` | Listar todas as áreas do tenant atual |
| `GET /api/v1/service-areas/active` | `AUTHENTICATED` | Listar somente áreas ativas |
| `GET /api/v1/service-areas/{id}` | `AUTHENTICATED` | Buscar área por ID |
| `POST /api/admin/service-areas` | `ADMIN` | Criar nova área de serviço |
| `PATCH /api/admin/service-areas/{id}/deactivate` | `ADMIN` | Desativar área |
| `PATCH /api/admin/service-areas/{id}/activate` | `ADMIN` | Reativar área |

> Endpoints em `/api/admin/**` não exigem `X-Tenant-ID` (bypass já configurado no `TenantResolverFilter`). A validação de tenant para essas rotas é feita via `@RequestParam tenantId` no corpo ou path.

### 2.2 Entregáveis por camada

#### Domain
- `ServiceAreaNotFoundException` em `domain/exceptions/` (mesmo padrão de `TripNotFoundException`)
- Método `rename(String newName)` em `ServiceArea` para suportar atualização de nome

#### Application
- `GetServiceAreaInputPort` — busca por ID
- `ListServiceAreasInputPort` — lista todas / somente ativas
- `CreateServiceAreaInputPort` — cria nova área (admin)
- `ToggleServiceAreaStatusInputPort` — ativa/desativa (admin)
- Use cases correspondentes em `application/usecases/servicearea/`

#### Infrastructure
- Cache Caffeine para `TenantRepositoryPort.existsById()` no `JpaTenantRepositoryAdapter`
  — implementar o TODO aberto desde HIST-2026-025; adicionar `tenantExistsCache` em `CacheConfiguration` (TTL 5 min)

#### Presentation

- `CreateServiceAreaRequest` (@NotBlank name, @NotBlank geoJsonPolygon)
- `ServiceAreaDtoMapper` (MapStruct, mesmo padrão dos outros mappers em `presentation/mappers/`)
- `ServiceAreaController` em `presentation/controllers/`
  — `@PreAuthorize("isAuthenticated()")` nos endpoints de leitura
  — `@PreAuthorize("hasRole('ADMIN')")` nos endpoints de escrita

## 5. Notas de Implementação

### Tenant em endpoints admin
Endpoints em `/api/admin/` não recebem `X-Tenant-ID` (bypass do filtro). O `CreateServiceAreaRequest` deve incluir `tenantId: UUID` no corpo para que o admin possa criar áreas para qualquer tenant. Internamente, o use case usa `TenantId` recebido no `Command`, não o `TenantContext`.

### Cache de tenant
Adicionar em `CacheConfiguration`:
```java
public static final String TENANT_EXISTS_CACHE = "tenantExists";
```
E em `JpaTenantRepositoryAdapter.existsById()`:
```java
@Cacheable(value = CacheConfiguration.TENANT_EXISTS_CACHE, key = "#tenantId.asString()")
```
TTL igual ao do `fareConfigByTenant` (30 min) ou menor (5 min). Adicionar `@CacheEvict` se `deleteTenant` for implementado.

### Domínio imutável
`ServiceArea` usa builder pattern sem setters. `rename()` deve seguir o mesmo padrão de `deactivate()`:
```java
public ServiceArea rename(String newName) {
    return new Builder().from(this).name(newName).updatedAt(Instant.now()).build();
}
```

### ArchUnit
O `ServiceAreaController` deve ficar em `presentation.controllers`, não em `presentation.controllers.admin` — a distinção admin/não-admin é feita pelos `@PreAuthorize` e pelo path (`/api/admin/**`), não pelo pacote.
