# HIST-2026-032: Compliance — CPF e Verificação de Telefone no Cadastro de Passageiros

**Data**: 05/05/2026
**Status**: 📋 Planejado
**Tipo**: Compliance / Security
**Tamanho**: Médio (2–3 dias)
**Prioridade**: Alta

---

## 1. Contexto & Motivação

A revisão de produto de maio/2026 identificou dois gaps de compliance no cadastro de passageiros que impactam conformidade fiscal, prevenção a fraudes e segurança da conta:

### Gap 1 — Ausência de CPF no cadastro

A entidade `Passenger` armazena apenas nome, e-mail e telefone. O CPF é ausente mesmo sendo:

- **Obrigatório para NF-e de transporte**: a Receita Federal exige CPF do tomador do serviço na nota fiscal eletrônica de serviços de transporte (NFS-e municipal).
- **Instrumento de anti-fraude**: sem CPF, o mesmo indivíduo pode criar múltiplas contas com e-mails diferentes sem nenhum controle de duplicidade real.
- **Padrão de mercado**: Uber, 99 e iFood coletam CPF no cadastro de usuários no Brasil.

O value object `CPF` já existe no domínio (usado pelo `Driver`) com validação completa de dígitos verificadores — pode ser reutilizado diretamente.

### Gap 2 — Conta ativa sem verificação de identidade

O `CreatePassengerUseCase` cria o passageiro com status `ACTIVE` imediatamente após o `POST /api/v1/passengers`. Não há:

- Verificação de e-mail (link de confirmação).
- Verificação de telefone (OTP via SMS).

Consequências práticas:
- Qualquer dado fake cria uma conta válida capaz de solicitar corridas.
- Um ator malicioso pode criar centenas de contas automatizadas (bots de fraude).
- Motoristas ficam expostos a passageiros sem identidade verificada.

**Decisão de arquitetura para OTP**: a verificação de telefone será implementada via **porta de saída `OtpServicePort`** com adapter inicial baseado em um **SMS gateway externo** (ex: AWS SNS, Twilio). O Keycloak não será usado para OTP de telefone — ele suporta TOTP (autenticador) mas não SMS OTP nativamente sem plugin pago. Isolar atrás de uma porta permite trocar o provider sem alterar o domínio.

## 2. Objetivo e Escopo

### 2.1 Adicionar CPF ao cadastro de passageiro

**Domain:**
- Adicionar campo `CPF cpf` à entidade `Passenger` — obrigatório no construtor de criação, presente no construtor de reconstrução.
- `PassengerRepositoryPort` recebe método `boolean existsByCpf(CPF cpf)`.

**Application:**
- `CreatePassengerInputPort.CreatePassengerCommand` passa a incluir `CPF cpf`.
- `CreatePassengerUseCase` valida unicidade de CPF antes de persistir (mesmo padrão do driver).

**Infrastructure:**
- Migration Flyway: `ALTER TABLE passengers ADD COLUMN cpf VARCHAR(11) NOT NULL` (nova coluna; migration deve ser executada antes do deploy — tabela nova em MVP, sem dados existentes para migrar).
- `PassengerJpaEntity`: adicionar campo `cpf` com `@Convert(converter = CPFConverter.class)` (converter já existe para driver).
- `SpringDataPassengerRepository`: adicionar `boolean existsByCpf(String cpf)`.
- `PassengerMapper` (MapStruct): mapear `cpf` nos dois sentidos.

**Presentation:**
- `CreatePassengerRequest`: adicionar `@NotBlank @Pattern(regexp = "\\d{11}") String cpf`.
- `PassengerResponse`: adicionar `String cpf` (formatado: `***.***.***-XX` para privacidade).

### 2.2 Verificação de telefone via OTP

**Fluxo após esta história:**

```
1. App → POST /api/v1/passengers          → cria conta em PENDING_VERIFICATION
2. Backend → SMS com código OTP de 6 dígitos (TTL: 10 min)
3. App → POST /api/v1/passengers/{id}/verify-phone  { "code": "123456" }
4. Backend valida código → muda status para ACTIVE
5. Passageiro pode solicitar corridas
```

**Novo status no domínio:**
- Adicionar `PENDING_VERIFICATION` ao enum `PassengerStatus` entre a criação e a ativação.
- `Passenger.verifyPhone()` → transição `PENDING_VERIFICATION → ACTIVE`. Lança `InvalidPassengerStateException` se status não for `PENDING_VERIFICATION`.
- `Passenger.canRequestRide()` retorna `false` para status `PENDING_VERIFICATION`.

**Nova porta de saída — `OtpServicePort`:**
```java
// application/ports/output/OtpServicePort.java
public interface OtpServicePort {
    void sendPhoneOtp(Phone phone, String code);
    // Validação é feita no use case (não delegar ao adapter para manter lógica no domínio)
}
```

**Armazenamento do OTP:**
- Não usar banco relacional para OTPs — usar **Redis** com TTL ou, para MVP sem Redis, a tabela `phone_verification_codes` (migration inclusa) com colunas: `id`, `passenger_id`, `code_hash` (BCrypt), `expires_at`, `used`.
- Output port: `PhoneVerificationRepositoryPort` com métodos `save(PhoneVerificationCode code)` e `findValidByPassengerId(UUID passengerId)`.

**Novos use cases:**
- `SendPhoneOtpUseCase` — gera código aleatório de 6 dígitos, hash BCrypt, persiste com TTL de 10 min, chama `OtpServicePort.sendPhoneOtp()`.
- `VerifyPhoneOtpUseCase` — busca código válido, valida hash, chama `passenger.verifyPhone()`, invalida o código usado.

**Endpoints:**
- `POST /api/v1/passengers/{id}/send-otp` — reenvio de código (rate-limited: máximo 3 envios por 30 min por passageiro).
- `POST /api/v1/passengers/{id}/verify-phone` — verificação do código.

**Adapter MVP:**
- `AwsSnsOtpAdapter` em `infrastructure/adapters/sms/` implementando `OtpServicePort`.
- Configurável via `application.properties`: `rappidrive.otp.provider=aws-sns` (permite troca futura por Twilio sem alterar use case).
- Em `test` profile: `MockOtpAdapter` que loga o código em vez de enviar SMS (evita custo em testes).

## 3. Arquitetura e Modelagem

### 3.1 Domain

**`PassengerStatus`** — novo valor:
```java
public enum PassengerStatus {
    PENDING_VERIFICATION,  // Aguardando verificação de telefone
    ACTIVE,
    INACTIVE,
    BLOCKED
}
```

**`Passenger`** — alterações:
```java
// Campo novo
private final CPF cpf;

// Construtor de criação — status inicia como PENDING_VERIFICATION
this.status = PassengerStatus.PENDING_VERIFICATION;

// Novo comportamento
public void verifyPhone() {
    if (status != PassengerStatus.PENDING_VERIFICATION) {
        throw new InvalidPassengerStateException("Cannot verify phone for passenger with status: " + status);
    }
    this.status = PassengerStatus.ACTIVE;
    this.updatedAt = LocalDateTime.now();
}

// Atualizar canRequestRide()
public boolean canRequestRide() {
    return status == PassengerStatus.ACTIVE;
}
```

**Novo value object `PhoneVerificationCode`** em `domain/valueobjects/`:
```java
public final class PhoneVerificationCode {
    private final UUID id;
    private final UUID passengerId;
    private final String codeHash;
    private final LocalDateTime expiresAt;
    private boolean used;

    public boolean isExpired() { return LocalDateTime.now().isAfter(expiresAt); }
    public void markAsUsed() { this.used = true; }
}
```

## 4. Critérios de Aceite

### CPF
- [ ] `POST /api/v1/passengers` sem `cpf` retorna 400.
- [ ] `POST /api/v1/passengers` com CPF inválido (dígitos verificadores errados) retorna 400.
- [ ] `POST /api/v1/passengers` com CPF já cadastrado retorna 409.
- [ ] `PassengerResponse` expõe CPF mascarado: `***.***.***-XX`.

### OTP e Verificação
- [ ] Passageiro criado com `POST /api/v1/passengers` tem status `PENDING_VERIFICATION`.
- [ ] SMS com código de 6 dígitos é enviado ao telefone cadastrado imediatamente após criação.
- [ ] `POST /passengers/{id}/verify-phone` com código correto e dentro do TTL muda status para `ACTIVE`.
- [ ] `POST /passengers/{id}/verify-phone` com código errado retorna 422.
- [ ] `POST /passengers/{id}/verify-phone` com código expirado retorna 422.
- [ ] Código usado não pode ser reutilizado (retorna 422 na segunda tentativa).
- [ ] `POST /passengers/{id}/send-otp` enviado mais de 3 vezes em 30 minutos retorna 429.
- [ ] Passageiro em `PENDING_VERIFICATION` não consegue solicitar corrida (retorna 403 em `POST /trips`).
- [ ] Em profile `test`/`dev`, código OTP é logado no console (não envia SMS real).

### Geral
- [ ] `mvn test` verde — ArchUnit + testes unitários de `SendPhoneOtpUseCase` e `VerifyPhoneOtpUseCase`.
- [ ] `mvn verify` verde — IT cobre criação de passageiro com CPF + verificação de OTP (mock do SMS).
