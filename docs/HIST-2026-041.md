# HIST-2026-041: Upgrade para Spring Boot 3.4.5 e Spring Cloud 2024.0.0

**Data**: 06 de Maio de 2026  
**Status**: ✅ Concluído  
**Tipo**: Infraestrutura / Manutenibilidade  
**Tamanho**: Médio (2 dias)

---

## 1. Contexto

A aplicação estava rodando com **Spring Boot 3.2.1**, que começou a apresentar limitações de compatibilidade com novas versões de bibliotecas e padrões de observabilidade de 2026. A migração para a **v3.4.5** garante suporte a longo prazo, logging estruturado nativo e melhorias de performance no Hibernate 6.6.

---

## 2. Mudanças Realizadas

### 2.1 Dependências (pom.xml)
- **Parent**: Atualizado para `org.springframework.boot:spring-boot-starter-parent:3.4.5`.
- **Spring Cloud**: Atualizado para a release train `2024.0.0` (Moorgate).
- **Flyway**: Adicionada a dependência modular `flyway-database-postgresql` exigida pela v10+.
- **SpringDoc**: Atualizado para `2.8.1` para suporte a novos padrões de roteamento.
- **Lombok & MapStruct**: Versões sincronizadas para Java 21 (`1.18.34` e `1.6.2`).

### 2.2 Unificação da Segurança
O padrão anterior de múltiplos beans `SecurityFilterChain` em classes de configuração separadas (`NoSecurityConfiguration`, `TestSecurityConfiguration`) causava conflitos de precedência no Spring Boot 3.4.

**Nova Abordagem:**
- Centralização em `SecurityConfiguration.java`.
- Uso de propriedades (`rappidrive.security.test-mode`) em vez de profiles complexos para alternar entre os modos de segurança.
- Garantia de mutualidade exclusiva entre os filtros via `@ConditionalOnExpression`.

### 2.3 Hibernate 6.6 e UUIDs Manuais
O Hibernate 6.6 alterou o comportamento de detecção de entidades "novas". Quando um ID (UUID) é atribuído manualmente na camada de domínio (nosso padrão), o uso de `@GeneratedValue` na entidade JPA causava `StaleObjectStateException` em operações de persistência.

**Solução:**
- Removido `@GeneratedValue` das entidades `DriverJpaEntity` e `PassengerJpaEntity`. O ID agora é tratado como puramente atribuído, respeitando a integridade do domínio.

### 2.4 Mapeamento de Records (Jackson)
Adicionado o argumento de compilação `-parameters` no `maven-compiler-plugin`. Isso garante que os nomes dos componentes de Java Records sejam preservados no bytecode, permitindo que o Jackson faça o mapeamento JSON ↔ Record sem a necessidade de anotações `@JsonProperty` redundantes.

## 3. Lições Aprendidas (Gotchas)

1.  **Conflito de Beans**: Spring Boot 3.4 é mais rigoroso com múltiplos beans do mesmo tipo que capturam o "match all" (`/**`). Sempre prefira unificar configurações de infraestrutura.
2.  **Flyway Modularity**: A partir da v10, o suporte a bancos específicos não está no core. Sempre verifique se o driver específico do banco está presente.
3.  **Persistência de IDs**: Ao usar Arquitetura Hexagonal onde o Domínio gera o ID, o Hibernate não deve tentar gerir a estratégia de geração.
